## 1. ОБЗОР

Мульти-модельная стратегия АСД v11.3 построена вокруг принципа **минимального
количества моделей при максимальном покрытии задач**. На Mac Studio M4 Max
с 128 GB Unified Memory одновременно работают всего **три уникальные LLM**
и модель embeddings. Такая конфигурация стала возможна благодаря архитектуре
**shared memory** — одна копия Gemma 4 31B обслуживает сразу пять рабочих
агентов, переключение между которыми осуществляется исключительно через
системные промпты без перезагрузки модели.

Переход с Qwen3.5-27B на Gemma 4 31B в v11.3 обусловлен тремя ключевыми
преимуществами: **нативная VLM/vision** (встроенный OCR без отдельной модели),
**128K контекст** (большинство договоров помещаются целиком без Map-Reduce) и
**лучшая модель для юридического анализа**. Делопроизводитель выделен на
отдельную лёгкую модель Gemma 4 E4B (~3 GB), поскольку его задачи (реестры,
архив, делопроизводство) не требуют тяжёлой модели с длинным контекстом.

### Ключевые принципы

1. **Постоянная загрузка:** Llama 3.3 70B, Gemma 4 31B и Gemma 4 E4B
   загружены всегда — нулевое время ожидания при переключении агентов.
2. **Shared memory:** Одна копия Gemma 4 31B обслуживает 5 агентов через
   системные промпты, экономя ~92 GB RAM (5 × 23 GB → 23 GB).
3. **4-bit квантизация:** Все LLM используют 4-bit квантизацию — оптимальный
   баланс качества и потребления RAM на Apple Silicon.
4. **Нативная VLM:** Gemma 4 31B (MLX-VLM) имеет встроенное зрение —
   OCR сканов и анализ чертежей без отдельной vision-модели.
5. **128K контекст:** Большинство типовых договоров (до ~500 страниц)
   помещаются в контекст целиком — Map-Reduce нужен только для ПД.
6. **OOM-защита:** Многоуровневая система контроля памяти с ограничением
   приёма задач и автоматической деградацией.
7. **cachetools вместо Redis:** Кэширование реализовано через cachetools
   (in-process, без внешних зависимостей) — проще, надёжнее, нет сетевых
   задержек.

параметрам: нативная VLM, 128K контекст, лучшее качество для юридического
анализа.

## 2. МОДЕЛИ

### 2.1. Llama 3.3 70B — Оркестратор (Hermes)

| Параметр          | Значение                                          |
|-------------------|---------------------------------------------------|
| **Параметров**    | 70 млрд                                           |
| **Квантизация**   | 4-bit                                             |
| **RAM**           | **~40 GB**                                        |
| **Контекст**      | до 128K tokens                                    |
| **Формат**        | MLX (.safetensors)                                |
| **Бэкенд**        | MLX (mac_studio) / Ollama (dev_linux)             |
| **Скорость**      | ~3–7 tokens/sec (M4 Max)                          |
| **Роль**          | Маршрутизация, координация, вердикты, PM-агент    |
| **Загрузка**      | Постоянная, при старте системы                    |

**Почему Llama 3.3 70B выбрана оркестратором:**

Llama 3.3 70B является наиболее мощной моделью в нашем арсенале и выполняет
критически важную роль координатора всей системы. Как оркестратор (агент PM /
Руководитель проекта), она отвечает за маршрутизацию входящих запросов к
соответствующим рабочим агентам, принимает финальные вердикты по результатам
анализа, управляет потоком задач между агентами и обеспечивает согласованность
результатов. Качество принятия решений на этом уровне критично для всей системы:
ошибка маршрутизации или неверный вердикт ведёт к потере качества downstream.
Именно поэтому для данной роли выбрана самая мощная из доступных моделей.
Модель загружена постоянно — это ядро системы, которое не выгружается ни при
каких обстоятельствах, кроме критического давления памяти.

### 2.2. Gemma 4 31B — Основная рабочая модель (VLM)

| Параметр          | Значение                                          |
|-------------------|---------------------------------------------------|
| **Параметров**    | 31 млрд                                           |
| **Квантизация**   | 4-bit                                             |
| **RAM**           | **~23 GB**                                        |
| **Контекст**      | до 128K tokens                                    |
| **Формат**        | MLX-VLM (.safetensors)                            |
| **Бэкенд**        | MLX-VLM (mac_studio) / Ollama (dev_linux)         |
| **Скорость**      | ~6–12 tokens/sec (M4 Max)                         |
| **Роль**          | 5 рабочих агентов через shared memory + VLM/OCR   |
| **Загрузка**      | Постоянная, при старте системы                    |
| **VLM**           | Да — нативная поддержка vision, встроенный OCR    |

**Ключевое преимущество — shared memory + VLM:**

Архитектура shared memory является центральным архитектурным решением АСД v11.3.
Вместо загрузки отдельной модели для каждого из пяти рабочих агентов (что
потребовало бы 5 × 23 GB = 115 GB RAM), одна копия Gemma 4 31B обслуживает все
пять агентов последовательно. Переключение между агентами происходит мгновенно
через подстановку соответствующего системного промпта — никакого ожидания
загрузки модели, никакого дополнительного расхода памяти.

В отличие от v11.2.2, где для OCR требовалась отдельная on-demand модель
(minicpm-v, ~5 GB), Gemma 4 31B имеет **нативную поддержку vision** через
MLX-VLM бэкенд. Это означает, что анализ сканов, чертежей и схем выполняется
той же моделью, что и текстовый анализ — без переключения моделей и без
дополнительного расхода RAM.

```

│       ▼              ▼              ▼                          │
│  ┌─────────────────────────────────────────────────────┐      │
│  │     Gemma 4 31B 4-bit (MLX-VLM)  ·  ~23 GB         │      │
│  │     (одна копия в памяти, 5 агентов + VLM)         │      │
│  └─────────────────────────────────────────────────────┘      │
```

**Агенты, использующие Gemma 4 31B (shared):**

| Агент                  | Ключ  | Роль в системе                                    |
|------------------------|-------|---------------------------------------------------|
| ПТО                    | pto   | Анализ проектной документации, проверка чертежей  |
| Сметчик                | smeta | Анализ смет, проверка расчётов, КОС              |
| Юрист                  | legal | Юридический анализ, экспертиза договоров, ФЗ      |
| Закупщик               | procurement | Закупочные процедуры, 44-ФЗ, 223-ФЗ       |
| Логист                 | logistics | Логистика, поставки, графики                  |

**Почему Gemma 4 31B вместо Qwen3.5-27B:**

Переход с Qwen3.5-27B на Gemma 4 31B в v11.3 обусловлен четырьмя критическими
преимуществами, каждое из которых непосредственно влияет на качество работы АСД:

1. **Нативная VLM / Vision:** Gemma 4 31B — мультимодальная модель с
   встроенным зрением через MLX-VLM бэкенд. Это устраняет необходимость в
   отдельной vision-модели (minicpm-v, ~5 GB) для OCR сканов и анализа
   чертежей. ПТО-агент теперь может анализировать сканированные чертежи и
   схемы той же моделью, что и текстовую документацию — без переключения
   моделей, без on-demand загрузки, без дополнительного расхода RAM.

2. **128K контекст:** Оба модели поддерживают 128K tokens, но Gemma 4 31B
   демонстрирует лучшее удержание внимания на длинных контекстах. Это
   критично для агента Юрист, анализирующего многостраничные договоры: при
   128K контексте (~96K слов, ~500 страниц) подавляющее большинство типовых
   договоров и смет помещаются целиком без Map-Reduce.

3. **Лучшее качество юридического анализа:** Gemma 4 31B показывает
   превосходные результаты в задачах юридического рассуждения (legal reasoning),
   особенно для русскоязычной правовой документации. Это подтверждено бенчмарками
   и внутренним тестированием на реальных договорах и тендерной документации.

4. **Единая модель для текста и vision:** Вместо двух моделей (Qwen3.5-27B
   для текста + minicpm-v для OCR) используется одна Gemma 4 31B — меньше
   моделей, проще управление памятью, нет задержек на загрузку/выгрузку
   on-demand моделей.

Дополнительный расход RAM (+8 GB по сравнению с Qwen3.5-27B) с лихвой
компенсируется устранением minicpm-v (~5 GB on-demand) и упрощением
архитектуры (нет on-demand загрузки, нет переключения моделей для OCR).

### 2.3. Gemma 4 E4B — Делопроизводитель

| Параметр          | Значение                                          |
|-------------------|---------------------------------------------------|
| **Параметров**    | 4 млрд                                            |
| **Квантизация**   | 4-bit                                             |
| **RAM**           | **~3 GB**                                         |
| **Контекст**      | до 8K tokens                                      |
| **Формат**        | MLX (.safetensors)                                |
| **Бэкенд**        | MLX (mac_studio) / Ollama (dev_linux)             |
| **Скорость**      | ~30–50 tokens/sec (M4 Max)                        |
| **Роль**          | Делопроизводитель — архив, реестры, делопроизводство |
| **Загрузка**      | Постоянная, при старте системы                    |

**Почему Делопроизводитель выделен на отдельную модель:**

В v11.3 Делопроизводитель (archive) переведён с Gemma 4 31B (shared) на
отдельную лёгкую модель Gemma 4 E4B. Это решение обусловлено двумя факторами:

1. **Характер задач:** Делопроизводитель выполняет относительно простые задачи —
   работа с архивом, ведение реестров, оформление делопроизводства. Эти задачи
   не требуют 128K контекста или глубокой юридической экспертизы. Контекста 8K
   tokens более чем достаточно для типичных операций архивариуса.

2. **Параллельная работа:** Выделение Делопроизводителя на отдельную модель
   позволяет ему работать параллельно с основными агентами на Gemma 4 31B.
   Пока пять рабочих агентов последовательно обрабатывают задачи через shared
   memory, Делопроизводитель может параллельно выполнять свои операции без
   ожидания в очереди.

Дополнительный расход: всего ~3 GB RAM. Это незначительная плата за
параллельность и освобождение слота в очереди shared memory.

### 2.4. bge-m3-mlx-4bit — Embeddings

| Параметр          | Значение                                          |
|-------------------|---------------------------------------------------|
| **Размерность**   | 1024                                              |
| **RAM**           | ~0.3 GB                                           |
| **Языки**         | 100+ (включая русский)                            |
| **Скорость**      | ~100 текстов/сек                                  |

bge-m3-mlx-4bit — MLX-оптимизированная 4-bit версия многоязычной модели

MLX 4-bit версию (~0.3 GB) экономит ~1.9 GB RAM при сохранении качества
векторного представления. Модель загружена постоянно, поскольку векторный
поиск используется практически в каждом запросе пользователя. Размерность
1024 обеспечивает хороший баланс между качеством представления и скоростью
поиска в PostgreSQL + pgvector.

### 3.2. Контекст (переменный)

Контекстная память расходуется динамически в зависимости от размера обрабатываемого
документа и текущей задачи. При 4-bit квантизации KV-кэш занимает существенно
меньше памяти, чем при fp16, что позволяет обрабатывать длинные документы без
риска OOM.

**Важное отличие v11.3:** Благодаря 128K контексту Gemma 4 31B, большинство
типовых договоров (до ~500 страниц) помещаются в контекст целиком. Map-Reduce
требуется только для проектной документации (2000+ страниц) и особо крупных
договоров.

| Операция                                | Доп. RAM    | Длительность       |
|-----------------------------------------|-------------|--------------------|
| Анализ договора целиком (Gemma 4 31B)   | +8–15 GB    | 30–90 сек          |
| Анализ чанка (Gemma 4 31B 4-bit)        | +5–10 GB    | 20–40 сек          |
| Map-Reduce (последовательно, 1 чанк)    | +5–10 GB    | ~25 сек/чанк       |
| Генерация DOCX                          | +1–2 GB     | 10–30 сек          |
| Маршрутизация (Llama 3.3 70B)          | +3–5 GB     | 5–15 сек           |
| VLM / OCR скана (Gemma 4 31B VLM)      | +2–5 GB     | 5–20 сек/страница  |
| Делопроизводитель (Gemma 4 E4B)        | +0.5–1 GB   | 5–15 сек           |

### 3.3. Пиковая нагрузка

Типичный сценарий пиковой нагрузки: анализ крупного документа через Map-Reduce
с одновременной генерацией отчёта и OCR отдельных сканов через VLM.

```
ПИКОВАЯ НАГРУЗКА — ХУДШИЙ СЦЕНАРИЙ
═══════════════════════════════════════════════════════════════

  Базово (модели):       66.3 GB   ← Llama + Gemma 4 31B + E4B + bge
  Система:               18.0 GB   ← macOS + PostgreSQL + App
  Map-Reduce пик:        10.0 GB   ← один чанк в обработке
  VLM / OCR:              5.0 GB   ← встроена в Gemma 4 31B (контекст)
  Генерация:              2.0 GB   ← финальная генерация DOCX
  ─────────────────────────────────
  ПИК:                  101.3 GB   ← 79.1% от 128 GB
  ОСТАЁТСЯ:              26.7 GB   ← запас 20.9%

═══════════════════════════════════════════════════════════════
```

**Важно:** Map-Reduce работает **строго последовательно** (max_parallel=1).
В памяти одновременно находится только один чанк. Это фундаментальное
архитектурное решение, гарантирующее, что пиковое потребление RAM от
Map-Reduce не превышает ~10 GB независимо от общего количества чанков.
Без этого ограничения параллельная обработка нескольких чанков могла бы
потребовать 30–50 GB дополнительной RAM.

**VLM не требует отдельной модели:** В отличие от v11.2.2, где OCR
требовал подгрузки minicpm-v (~5 GB on-demand), в v11.3 Gemma 4 31B
выполняет OCR тем же экземпляром модели. Дополнительный расход RAM
ограничивается лишь KV-кэшем для изображения (~2–5 GB), который
автоматически освобождается после обработки.

## 4. УРОВНИ ДАВЛЕНИЯ ПАМЯТИ

Базовая загрузка моделей: ~66 GB (52% от 128 GB). С учётом системных процессов
базовое потребление составляет ~84 GB (66%). Пороги настроены с учётом этого
базового уровня и обеспечивают многоуровневую защиту от OOM.

### 4.1. low (< 70%, < 90 GB)

Нормальный режим работы. Все основные модели загружены, система работает в
штатном порядке. VLM/OCR через Gemma 4 31B доступна без ограничений.
Последовательная обработка чанков Map-Reduce выполняется без ограничений.
Это целевой режим, в котором система проводит подавляющее большинство времени.

- Все основные модели загружены (Llama 70B + Gemma 4 31B + Gemma 4 E4B + bge-m3)
- VLM/OCR через Gemma 4 31B доступна без ограничений
- Последовательная обработка без ограничений
- Логирование: стандартный уровень

### 4.2. medium (70–80%, 90–102 GB)

Повышенная нагрузка. Основные модели остаются загружены, но система начинает
контролировать расход памяти более внимательно. Пик контекста при Map-Reduce
укладывается в бюджет, однако дальнейший рост потребления нежелателен.
Система автоматически выводит предупреждения в логи и готовится к возможному
переходу на уровень high.

- Все основные модели остаются загружены
- Пик контекста при Map-Reduce укладывается в бюджет
- VLM/OCR продолжает работу, но крупные изображения обрабатываются с осторожностью
- Предупреждение в логах при приближении к 80%
- Автоматический мониторинг каждые 15 секунд (вместо 30)

### 4.3. high (80–90%, 102–115 GB)

Высокое давление памяти. Система переходит в защитный режим: VLM/OCR задачи
с крупными изображениями блокируются, текстовый анализ через Gemma 4 31B
продолжает работать. Текущие задачи завершаются последовательно, но новые
задачи с высоким потреблением памяти ставятся в очередь. Пользователь
получает предупреждение о высоком использовании памяти. Если давление не
снижается в течение 60 секунд, система инициирует принудительную сборку
мусора.

- VLM/OCR с крупными изображениями блокируется
- Текстовый анализ и маршрутизация продолжают работу
- Оставшиеся задачи завершаются последовательно
- Предупреждение пользователю: «Высокое использование памяти»
- Новые задачи с высоким потреблением ставятся в очередь
- Мониторинг каждые 10 секунд

### 4.4. critical (> 90%, > 115 GB)

Критическое давление памяти. Система переходит в режим аварийной защиты:
приём новых задач полностью останавливается. Выполняются только текущие
задачи, которые будут завершены в минимальном режиме. Если критический
уровень сохраняется более 30 секунд, рассматривается выгрузка Gemma 4 31B
(экономия 23 GB) — при этом все 5 рабочих агентов и VLM становятся
недоступны до перезагрузки модели. Gemma 4 E4B может быть выгружена
дополнительно (экономия 3 GB).

- Новые задачи отклоняются с ошибкой `MEMORY_CRITICAL`
- Пользователь: «Критическое использование памяти. Подождите завершения текущих задач.»
- При длительном критическом уровне (> 30 сек) — выгрузка Gemma 4 31B
  (экономия 23 GB, но все 5 рабочих агентов + VLM недоступны)
- При экстремальном уровне — выгрузка Gemma 4 E4B (доп. экономия 3 GB)
- Только агент PM (Llama 3.3 70B) остаётся полностью функциональным
- Мониторинг каждые 5 секунд
- Автоматическое восстановление при снижении давления до medium

### 5.2. Fallback-цепочки

Fallback-цепочки определяют порядок деградации при недоступности основной
модели. Каждая цепочка обеспечивает максимально возможное качество при
постепенном снижении доступных ресурсов.

```
FALLBACK-ЦЕПОЧКИ v11.3
═══════════════════════════════════════════════════════════════════

1. Оркестратор недоступен:
   Llama 3.3 70B  ──✕──▶  Gemma 4 31B (сниженное качество,
                               работает как текстовый маршрутизатор)
                         ──✕──▶  Ошибка «Оркестратор недоступен»

2. Основная рабочая модель недоступна:
   Gemma 4 31B  ──✕──▶  Llama 3.3 70B (сниженное качество,
                              повышенный расход RAM на контекст,
                              VLM/OCR недоступен)
                    ──✕──▶  Ошибка «Рабочая модель недоступна»

3. Делопроизводитель недоступен:
   Gemma 4 E4B  ──✕──▶  Gemma 4 31B (через shared memory,
                               избыточно для простых задач)
                    ──✕──▶  Llama 3.3 70B (крайний fallback)

4. Embeddings недоступны:
   bge-m3-mlx-4bit  ──✕──▶  Полнотекстовый поиск (pg_trgm)
                     ──✕──▶  Ошибка «Поиск временно недоступен»

5. VLM / OCR недоступен (Gemma 4 31B без изображений):
   Gemma 4 31B VLM  ──✕──▶  pytesseract (базовый OCR, низкое кач.)
                    ──✕──▶  Ошибка «OCR недоступен»

═══════════════════════════════════════════════════════════════════
```

**Важные замечания по fallback:**

- При fallback Gemma 4 31B → Llama 3.3 70B для рабочих агентов качество
  остаётся приемлемым (Llama 70B мощнее), но время отклика увеличивается
  (3–7 tok/s вместо 6–12 tok/s), а VLM/OCR становится недоступен —
  сканы и чертежи не могут быть обработаны.
- При fallback оркестратора на Gemma 4 31B качество маршрутизации снижается,
  но система остаётся функциональной — Gemma 4 31B способна выполнять
  базовую маршрутизацию, хотя вердикты будут менее точными.
- При fallback Делопроизводителя на Gemma 4 31B задачи выполняются корректно,
  но увеличивается нагрузка на shared memory и время ожидания в очереди.
- Fallback embeddings на pg_trgm обеспечивает работоспособность поиска,
  но без семантического понимания — только лексическое совпадение.

## 6. MAP-REDUCE СТРАТЕГИЯ

### 6.1. Почему Map-Reduce (и когда он НЕ нужен)

Gemma 4 31B имеет контекст до 128K tokens (~96K слов, ~500 страниц). Благодаря
этому **большинство типовых документов помещаются в контекст целиком**, и
Map-Reduce для них не требуется. Это ключевое преимущество v11.3 по сравнению
с архитектурами на моделях с 32K контекстом, где Map-Reduce нужен был даже
для 30-страничных договоров.

**Когда Map-Reduce НЕ нужен:**
- Типовые договоры (до ~500 страниц / 128K tokens)
- Сметы и КОС (до ~500 страниц)
- Закупочная документация (до ~500 страниц)
- Отдельные разделы ПД

**Когда Map-Reduce нужен:**
- Полная проектная документация (2000+ страниц)
- Комплексные тендерные пакеты (1500+ страниц)
- Любой документ, превышающий ~128K tokens

Для документов реального объёма (ПД на 2000+ страниц) Map-Reduce остаётся
единственным практически применимым подходом по трём причинам:

1. **Качество анализа:** Модель значительно лучше анализирует чанки среднего
   размера (6000 tokens), чем документ целиком. При большом контексте модель
   «размывает» внимание и пропускает детали. Чанк 6000 tokens даёт достаточный
   контекст для понимания юридических конструкций, но не перегружает модель.

2. **Экономия памяти:** При 4-bit квантизации KV-кэш для полного контекста
   128K tokens может занимать до 15+ GB. Последовательная обработка чанков
   по 6000 tokens ограничивает пиковое потребление ~10 GB, что критично для
   128 GB Unified Memory.

3. **Рекурсивный Reduce:** Многораундовая агрегация (3 раунда) даёт более
   точный финальный результат, чем попытка «сжать» всё в один проход.
   Каждый раунд Reduce консолидирует результаты предыдущего, постепенно
   формируя целостную картину.

### 6.2. Параметры

| Параметр           | Значение       | Обоснование                                         |
|--------------------|----------------|-----------------------------------------------------|
| Chunk size         | **6000 tokens**| Больше контекста, точнее анализ юридических конструкций |
| Overlap            | **300 tokens** | Минимизация потерь на границах чанков              |
| Max parallel       | **1**          | Последовательная обработка, минимальный пик RAM    |
| Timeout на чанк    | 300 сек        | 5 минут на чанк — достаточно для сложного анализа  |
| Max retries        | 2              | Повтор при ошибке LLM                              |
| Reduce rounds      | **3**          | Map → Reduce1 (×10) → Reduce2 (×10) → Final       |
| Модель Map         | **Gemma 4 31B**| Основная рабочая модель                            |
| Модель Reduce      | **Gemma 4 31B**| Та же модель (shared, без доп. расхода RAM)        |
| Модель финального  | **Llama 3.3 70B**| Оркестратор формирует итоговый вердикт           |
| вердикта           |                |                                                     |

## 7. MLX КОНФИГУРАЦИЯ (mac_studio)

На Mac Studio все LLM-модели работают через **MLX / MLX-VLM** бэкенд —
фреймворк Apple для машинного обучения на Apple Silicon. MLX обеспечивает
нативный доступ к Unified Memory, Metal GPU и Neural Engine, что даёт
значительное преимущество в производительности по сравнению с Ollama
на macOS. MLX-VLM добавляет поддержку vision-моделей с теми же
преимуществами.

### 7.1. Модели в config.py (mac_studio профиль)

Конфигурация моделей задана в `PROFILE_MODELS` (config.py) и автоматически
выбирается через `settings.get_model_config(agent_name)` на основе текущего
профиля. Shared memory реализована через указание одного и того же пути к
модели для нескольких агентов — MLX загружает модель один раз и использует
её для всех запросов.

```
┌──────────────┬──────────────────────────────────────────────┬──────────┐
│ Агент        │ MLX Model                                    │ RAM      │
├──────────────┼──────────────────────────────────────────────┼──────────┤
│ pm           │ mlx-community/Meta-Llama-3.3-70B-Instruct-4bit│ ~40 GB │
│ pto          │ mlx-community/gemma-4-31b-it-4bit (VLM)      │ ~23 GB   │
│ smeta        │ mlx-community/gemma-4-31b-it-4bit (VLM)      │ ~23 GB   │
│ legal        │ mlx-community/gemma-4-31b-it-4bit (VLM)      │ ~23 GB   │
│ procurement  │ mlx-community/gemma-4-31b-it-4bit (VLM)      │ ~23 GB   │
│ logistics    │ mlx-community/gemma-4-31b-it-4bit (VLM)      │ ~23 GB   │
│ archive      │ mlx-community/gemma-4-e4b-it-4bit            │ ~3 GB    │
│ embed        │ bge-m3-mlx-4bit                              │ ~0.3 GB  │
├──────────────┼──────────────────────────────────────────────┼──────────┤
│ ИТОГО        │                                              │ 66.3 GB  │
│ (постоянно)  │  Llama + Gemma 4 31B(shared) + E4B + bge    │          │
└──────────────┴──────────────────────────────────────────────┴──────────┘
```

**Shared memory — детали реализации:**

- Gemma 4 31B загружается **один раз** при старте системы — обслуживает
  pto, smeta, legal, procurement, logistics + VLM/OCR
- Переключение агента = подстановка системного промпта — мгновенно, без
  перезагрузки модели
- Llama 3.3 70B загружена постоянно — обслуживает pm
- Gemma 4 E4B загружена постоянно — обслуживает archive (параллельно)
- bge-m3-mlx-4bit работает через MLX — обслуживает embed
- VLM встроена в Gemma 4 31B — OCR без on-demand загрузки

### 7.2. Профиль и переключение

LLMEngine автоматически определяет бэкенд на основе переменной окружения
`ASD_PROFILE`:

```bash
# Mac Studio — продуктивная конфигурация
export ASD_PROFILE=mac_studio
# → LLM через MLX/MLX-VLM, embeddings через MLX, VLM встроена

# Linux — конфигурация разработки
export ASD_PROFILE=dev_linux
# → Все модели через Ollama
```

Конфигурация моделей и бэкендов задана в `PROFILE_MODELS` (config.py) и
выбирается через `settings.get_model_config(agent_name)`. При смене профиля
необходим перезапуск приложения для выгрузки/загрузки моделей.

### 7.3. Преимущества MLX на Mac Studio

MLX / MLX-VLM являются оптимальным выбором для работы LLM на Mac Studio
M4 Max по ряду причин:

- **Нативная работа с Unified Memory:** Модели видят все 128 GB как единую
  адресуемую память — нет разделения на VRAM и RAM, нет необходимости
  перемещать данные между устройствами. Это фундаментальное преимущество
  Apple Silicon.

- **Оптимизация под Apple Silicon:** MLX использует Metal GPU для матричных
  операций и Neural Engine для специфических вычислений. На M4 Max это даёт
  ~2–3× ускорение по сравнению с CPU-only inference.

- **Прямой доступ к памяти:** В отличие от Ollama, который выступает прослойкой
  между моделью и оборудованием, MLX работает напрямую с Unified Memory —
  меньше накладных расходов, точнее контроль потребления RAM.

- **MLX-VLM для vision:** MLX-VLM обеспечивает нативную поддержку
  мультимодальных моделей (Gemma 4 31B) на Apple Silicon — обработка
  изображений через тот же бэкенд, что и текстовая генерация.

- **Автоматическое управление KV-кэшем:** MLX framework автоматически управляет
  кэшем ключ-значение, освобождая память от контекста завершённых запросов.
  Это предотвращает утечки памяти при длительной работе системы.

### 8.2. Частота проверки

Частота мониторинга памяти адаптируется к текущему уровню давления: чем выше
давление, тем чаще проверяется состояние памяти. Это позволяет своевременно
реагировать на угрозу OOM и предотвращать аварийные ситуации.

| Событие                          | Действие                               |
|----------------------------------|----------------------------------------|
| Каждая новая задача              | Проверка перед запуском                |
| Каждые 30 сек (low)              | Фоновая проверка                       |
| Каждые 15 сек (medium)           | Учащённая проверка                     |
| Каждые 10 сек (high)             | Интенсивный мониторинг                 |
| Каждые 5 сек (critical)          | Непрерывный мониторинг                 |
| Завершение задачи                | Пересчёт доступной памяти              |
| Загрузка/выгрузка модели         | Обновление models_memory               |
| Давление = high                  | Предупреждение в лог                   |
| Давление = critical              | Отказ новых задач, уведомление         |

## 9. ОПТИМИЗАЦИЯ ПРОИЗВОДИТЕЛЬНОСТИ

### 9.1. Кэширование (cachetools)

Стратегия кэширования в АСД v11.3 построена на библиотеке **cachetools** —
in-process кэшировании без внешних зависимостей. Переход с Redis на cachetools
устраняет сетевые задержки, упрощает развёртывание и снижает потребление RAM
(нет отдельного процесса Redis).

| Что кэшировать               | Где                     | TTL       | Обоснование             |
|-------------------------------|-------------------------|-----------|-------------------------|
| Embeddings чанков             | PostgreSQL (vector)     | Постоянно | Текст не меняется       |
| Результаты поиска             | cachetools (TTLCache)   | 5 мин     | БД может обновляться    |
| Ответы LLM (идентичные запр.) | cachetools (TTLCache)  | 1 мин     | Только в рамках сессии  |
| Статус системы                | cachetools (TTLCache)   | 30 сек    | Динамические данные     |
| Системные промпты агентов     | cachetools (LRUCache)   | Постоянно | Статичные данные        |

**Почему cachetools вместо Redis:**

- **Нет внешних зависимостей:** Не нужен отдельный процесс Redis, нет сетевых
  задержек, нет проблем с подключением.
- **Простота развёртывания:** Одно приложение — один процесс, меньше движущихся
  частей.
- **Меньше RAM:** Нет отдельного процесса Redis (~50–200 MB в зависимости от
  объёма кэша).
- **Достаточно для АСД:** Система работает на одной машине, один пользователь
  за раз — распределённый кэш не требуется.

### 9.2. Батчинг embeddings

Батчинг — ключевой инструмент оптимизации при индексации больших документов.
Вместо последовательных запросов к bge-m3-mlx-4bit для каждого чанка, чанки
объединяются в батчи и обрабатываются за один вызов модели.

```python
# ─── Плохо: 500 запросов по 300 мс = 150 сек ───
for chunk in chunks:
    embedding = embed_model.encode(chunk.text)

# ─── Хорошо: батч по 32 = ~16 запросов по 1 сек = ~16 сек ───
batch_size = 32
for i in range(0, len(chunks), batch_size):
    batch = chunks[i:i+batch_size]
    embeddings = embed_model.encode([c.text for c in batch])
```

При батчинге важно контролировать размер группы: слишком большой батч
может вызвать скачок потребления RAM. Оптимальный размер — 32 чанка,
что соответствует ~192K tokens на батч.

### 9.3. Последовательный Map-Reduce

Последовательная обработка чанков — фундаментальное архитектурное решение,
гарантирующее предсказуемое потребление RAM. Вместо параллельной обработки
нескольких чанков (которая потребовала бы кратного увеличения контекстной
памяти), каждый чанк обрабатывается строго последовательно. После обработки
результат сохраняется, контекст освобождается, и только затем начинается
обработка следующего чанка. RAMManager проверяет состояние памяти после
каждого чанка.

```python
# ─── Последовательная обработка (4-bit, минимальный пик RAM) ───
async def analyze_chunk(chunk, agent_prompt):
    return await mlx_vlm.chat(
        "gemma-4-31b-it-4bit",
        messages=[
            {"role": "system", "content": agent_prompt},
            {"role": "user", "content": chunk.text}
        ]
    )

# Без параллелизма — один чанк за раз
results = []
for chunk in chunks:
    result = await analyze_chunk(chunk, agent_prompt)
    results.append(result)
    # RAMManager.check() после каждого чанка
```

### 9.4. Целый контекст вместо Map-Reduce

Ключевая оптимизация v11.3: **документы до ~128K tokens анализируются целиком**,
без разбиения на чанки. Это даёт три преимущества:

1. **Скорость:** Один вызов модели вместо Map-Reduce (30+ вызовов для
   150-страничного договора). Ускорение ~4–5× для типовых документов.
2. **Качество:** Модель видит весь документ целиком — нет потерь на границах
   чанков, нет искажений от агрегации Reduce.
3. **Меньше RAM:** Нет промежуточных результатов Map-Reduce, нет накладных
   расходов на агрегацию.

```python
# ─── Анализ договора целиком (если помещается в 128K) ───
if token_count(document) < 120_000:  # запас 8K на промпт
    result = await mlx_vlm.chat(
        "gemma-4-31b-it-4bit",
        messages=[
            {"role": "system", "content": legal_prompt},
            {"role": "user", "content": document_text}
        ]
    )
else:
    # Map-Reduce только для документов, не помещающихся в контекст
    result = await map_reduce_analyze(document, legal_prompt)
```

### 9.5. VLM / OCR через Gemma 4 31B

Встроенная VLM Gemma 4 31B обрабатывает изображения (сканы, чертежи, схемы)
без переключения моделей:

```python
# ─── OCR скана через VLM (без on-demand загрузки) ───
result = await mlx_vlm.chat(
    "gemma-4-31b-it-4bit",
    messages=[
        {"role": "system", "content": "Извлеки текст из скана документа."},
        {"role": "user", "content": [
            {"type": "image", "image": scan_image_base64},
            {"type": "text", "text": "Распознай текст и структуру таблицы."}
        ]}
    ]
)
```

Это устраняет задержки на загрузку/выгрузку minicpm-v и экономит ~5 GB RAM
в штатном режиме.
