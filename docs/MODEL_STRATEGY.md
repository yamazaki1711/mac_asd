## 2. МОДЕЛИ

### 2.1. Gemma 4 31B — основная LLM (СТАНДАРТ: FP16)

| Параметр | Значение |
|----------|----------|
| **Параметров** | 31 млрд |
| **Квантизация** | **FP16 (16-bit, ~62 GB)** — стандарт |
| **RAM** | **~62 GB** |
| **Контекст** | до 128K tokens |
| **Формат** | GGUF (llama.cpp) |
| **Бэкенд** | Metal (MPS) |
| **Скорость** | ~5-10 tokens/sec |

**Когда Q8_0 (~33 GB) вместо FP16:**

- Экстренная нехватка RAM (критическое давление памяти)
- Требуется параллельная работа с несколькими тяжёлыми задачами
- Быстрая проверка гипотезы перед полным анализом

**Почему FP16 по умолчанию:**

- Юридическая экспертиза: каждая формулировка на вес. Потеря 0.1-0.5% при квантизации = потенциально пропущенная ловушка
- Генерация иска/претензии: точность ссылок на нормы критична для суда
- Анализ ПД: коллизии и неучтённые объёмы — цена ошибки в миллионах
- Скорость не критична: 3-4 часа АСД vs неделя команды из 2-3 человек (40-50× ускорение)

## 3. RAM БЮДЖЕТ (128GB)

### 3.1. Базовая конфигурация (FP16, модели загружены)

| Компонент | RAM | Примечание |
|-----------|-----|------------|
| gemma-4-31b:fp16 | 62.0 GB | **Всегда загружена** — стандарт |
| bge-m3 | 2.2 GB | Всегда загружена |
| **Итого модели** | **64.2 GB** | |
| macOS | 10.0 GB | Система + сервисы |
| PostgreSQL | 3.0 GB | Кэш, буферы |
| АСД приложение | 5.0 GB | Python, зависимости |
| **Итого система** | **18.0 GB** | |
| **Занято базово** | **82.2 GB** | |
| **Свободно** | **45.8 GB** | На контекст и пиковые нагрузки |

### 3.2. Контекст (переменный)

| Операция | Доп. RAM | Длительность |
|----------|----------|--------------|
| Анализ чанка (31B FP16) | +5-10 GB | 20-40 сек |
| Map-Reduce (последовательно) | +5-10 GB (пик) | ~20 сек/чанк |
| Генерация DOCX | +1-2 GB | 10-30 сек |
| OCR скана (minicpm-v, подгружается) | +5 GB | 3-10 сек/стр |
| Подгрузка E4B | +2.5 GB | ~30 сек |

### 3.3. Пиковая нагрузка

Типичный сценарий: последовательный Map-Reduce анализа 650 чанков ПД

- подгруженный minicpm-v для OCR + генерация отчёта:

```
Базово (FP16):     82.2 GB
Map-Reduce пик:    10.0 GB (последовательно, один чанк)
OCR (minicpm-v):    5.0 GB (подгружается по необходимости)
Генерация:          2.0 GB
─────────────────────────
Пик:               99.2 GB
Остаётся:          28.8 GB
```

**Важно:** Map-Reduce работает **последовательно** (не параллельно).
Нет пика от 3 одновременных задач. RAM растёт плавно.

### 3.4. Альтернатива: Q8_0 при нехватке памяти

Когда нужно освободить RAM (давление > high):

```
gemma-4-31b:q8_0   33.0 GB  ← экономия 29 GB
bge-m3              2.2 GB
Система            18.0 GB
─────────────────────────
Базово:            53.2 GB
Контекст пик:      10.0 GB
Остаётся:          64.8 GB  ← запас для параллельных задач

E4B и minicpm-v подгружаются по необходимости.
Q8_0 — это emergency fallback, не стандарт.
```

## 4. УРОВНИ ДАВЛЕНИЯ ПАМЯТИ

Базовая загрузка при FP16: ~82 GB (64%). Поэтому пороги смещаются.

### 4.1. low (< 70% использовано, < 90 GB)

- FP16 31B + bge-m3 загружены
- Последовательная обработка (без параллелизма)
- minicpm-v и E4B подгружаются по необходимости, выгружаются после использования

### 4.2. medium (70-80%, 90-102 GB)

- FP16 31B остаётся
- Нормально: пик контекста при Map-Reduce укладывается
- Предупреждение в логах при приближении к 80%

### 4.3. high (80-90%, 102-115 GB)

- Все дополнительные модели (E4B, minicpm-v) немедленно выгружаются
- Оставшиеся задачи завершаются последовательно
- Предупреждение пользователю: "Высокое использование памяти"
- Новые задачи ставятся в очередь

### 4.4. critical (> 90%, > 115 GB)

- Переключение на Q8_0 (экономия 29 GB)
- Все дополнительные модели выгружаются
- Новые задачи отклоняются с ошибкой `MEMORY_CRITICAL`
- Пользователь: "Критическое использование памяти. Подождите завершения текущих задач."

## 6. MAP-REDUCE СТРАТЕГИЯ

### 6.1. Почему Map-Reduce?

Gemma 4 31B имеет контекст до 128K tokens (~96K слов, ~500 страниц).
Но:

1. **Качество:** Модель лучше анализирует чанки среднего размера (6000 tokens) — видит больше контекста, чем 4000, но не перегружена
2. **Память:** При FP16 большой контекст = большой расход RAM. Последовательная обработка минимизирует пик
3. **Рекурсивный Reduce:** Многораундовая агрегация даёт более точный финальный результат

### 6.2. Параметры

| Параметр | Значение | Почему |
|----------|----------|--------|
| Chunk size | **6000 tokens** | Больше контекста для FP16, точнее анализ юридических конструкций |
| Overlap | **300 tokens** | Больше overlap — меньше потерь на границах |
| Max parallel | **1** | Последовательная обработка, минимальный пик RAM |
| Timeout на чанк | 300 сек | 5 мин на чанк |
| Max retries | 2 | При ошибке LLM |
| Reduce rounds | **3** | Map → Reduce1 (×10) → Reduce2 (×10) → Final |

### 6.3. Расчёт времени

```
Договор 150 страниц:
  → ~185,000 слов
  → ~31 чанк (по 6000 tokens)
  → Последовательно: 31 × 25 сек = ~13 мин (Map)
  → Reduce (3 раунда): 31 → 4 → 1 = ~3 мин
  → Итого: ~16 мин

ПД 2450 страниц:
  → ~3,000,000 слов
  → ~500 чанков (по 6000 tokens)
  → Последовательно: 500 × 25 сек = ~208 мин ≈ 3.5 часа (Map)
  → Reduce (3 раунда): 500 → 50 → 5 → 1 = ~15 мин
  → Итого: ~3.5-4 часа

Человек: 5 дней × 3 человека = 15 человеко-дней
АСД: 3.5-4 часа непрерывной работы
Ускорение: ~40-50×
```

## 9. ОПТИМИЗАЦИЯ ПРОИЗВОДИТЕЛЬНОСТИ

### 9.1. Кэширование

| Что кэшировать | Где | TTL |
|----------------|-----|-----|
| Embeddings чанков | PostgreSQL (vector column) | Постоянно |
| Результаты поиска | Redis / in-memory | 5 мин |
| Ответы LLM (одинаковые запросы) | in-memory | 1 мин |
| Статус системы | in-memory | 30 сек |

### 9.2. Батчинг embeddings

Вместо последовательных запросов:

```python
# Плохо: 500 запросов по 300 мс = 150 сек
for chunk in chunks:
    embedding = ollama.embeddings("bge-m3", chunk.text)

# Хорошо: батч по 32 = ~16 запросов по 1 сек = ~16 сек
batch_size = 32
for i in range(0, len(chunks), batch_size):
    batch = chunks[i:i+batch_size]
    embeddings = ollama.batch_embeddings("bge-m3", [c.text for c in batch])
```

### 9.3. Последовательный Map-Reduce

```python
# Последовательная обработка (FP16, минимальный пик RAM)
async def analyze_chunk(chunk):
    return await ollama.chat("gemma-4-31b:fp16", messages, thinking=True)

# Без параллелизма — один чанк за раз
results = []
for chunk in chunks:
    result = await analyze_chunk(chunk)
    results.append(result)
    # RAMManager.check() после каждого чанка
    if RAMManager.should_unload():
        await ollama.stop("minicpm-v")  # освободить память
```
