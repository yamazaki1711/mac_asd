## 1. Назначение

`ppr_generator` — модуль генерации **Проекта производства работ (ППР)** в составе единого конвейера исполнительной документации АСД. ППР — это комплект документов, регламентирующий технологию, организацию и последовательность производства строительно-монтажных работ на объекте.

### Позиционирование в едином конвейере ИД

ППР — один из 13+ типов документов в полном комплекте исполнительной документации по Приказу 344/пр. Он не является якорным документом (якорь — АОСР) и не блокирует оплату при отсутствии, но часто требуется для комплектности.

Архитектура единого конвейера описана в `docs/id_pipeline_architecture.md`. Ключевая связь `ppr_generator` с конвейером:

- **TTK Registry** из `ppr_generator` используется `AOSRChainBuilder` для построения цепочек актов освидетельствования — ТТК определяет технологическую последовательность, по которой выстраиваются АОСР
- **Данные о качестве** из ТТК (допуски, методы контроля) используются при генерации АВК и протоколов лабораторных испытаний
- **Данные о ресурсах** (бригады, техника) используются при сборке КС-2/КС-3
- ППР генерируется по инициативе `InventoryEngine`, если обнаружена лакуна `doc_type="ppr"`

В отличие от `is_generator`, который генерирует один тип документа (Исполнительная схема — ИС), `ppr_generator` генерирует **полный комплект документов** — пояснительную записку и графическую часть, объединённые в единый PDF/DOCX.

### Чем ППР принципиально отличается от ИС

| Аспект | ИС (is_generator) | ППР (ppr_generator) |
|--------|--------------------|----------------------|
| **Результат** | Один документ (чертёж ИС) | Комплект документов (ПЗ + графика) |
| **Объём** | 1–3 страницы | 50–200+ страниц |
| **Вход** | DXF/DWG + данные контроля | ПОС + ПД/РД |
| **Формат вывода** | PDF (векторный чертёж) | PDF (документ) + DOCX (ПЗ) + DXF (чертежи) |
| **Структура** | Фиксированная (штамп + отклонения) | Модульная (12+ разделов, N технологических карт) |
| **Вариативность** | Низкая (один шаблон) | Высокая (состав разделов зависит от видов работ) |
| **Графика** | Схема с отклонениями | Стройгенплан, техсхемы, строповки, график Ганта |

## 2. Нормативная база

ППР разрабатывается в соответствии со следующими нормативными документами:

| Документ | Описание |
|----------|----------|
| **СНиП 12-01-2004** | Организация строительства |
| **СП 48.13330.2019** | Организация строительства (актуализированная редакция СНиП 12-01-2004) |
| **ГОСТ 21.1101-2013** | Основные требования к проектной и рабочей документации |
| **РД-11-02-2006** | Требования к составу и порядку ведения исполнительной документации |
| **ПОС** | Проект организации строительства — исходный документ для ППР |
| **Приказ Ростехнадзора от 26.12.2012 № 781** | Порядок проведения ПНР |

ППР разрабатывается на основе ПОС и является его детализацией для конкретного подрядчика. Если ПОС определяет «что и когда строить», то ППР определяет «как именно строить» — технологию, методы, ресурсы, безопасность.

## 3. Состав комплекта ППР

На основе анализа реального ППР (причал, шпунтовая стенка, 95 страниц) комплект состоит из двух частей:

### 3.1. Пояснительная записка (текстовая часть)

| № | Раздел | Обозначение | Содержание | Источник данных |
|---|--------|-------------|------------|-----------------|
| — | **Титульный лист** | — | Наименование объекта, заказчик, подрядчик, шифр, утверждающий | Входные параметры |
| — | **Лист согласования** | — | Подписи согласующих лиц | Входные параметры |
| 1 | **Общие данные** | ОД | Характеристика объекта, исходные данные, нормативная база, сроки строительства | ПОС |
| 2 | **Организация производства работ** | ОПР | Подготовительный период, решения по стройплощадке, ПРР, геодезия, зимние работы | ПОС + РД |
| 3 | **Технологические карты (ТТК/ТК)** | ТТК | Область применения → технология → качество → ресурсы. **По одному на каждый вид работ** | РД (виды работ + объёмы) |
| 4 | **Технология работ с ПС** | ПС | Общие положения, производство работ с грузоподъёмными механизмами | ТТК + РД (характеристики ПС) |
| 5 | **Контроль качества** | КК | Входной, операционный, приёмочный контроль | ТТК (критерии приёмки) |
| 6 | **Потребность в рабочих кадрах** | РК | Таблица: профессия, разряд, количество | ТТК (состав бригад) + ПОС (сроки) |
| 7 | **Потребность в технике и механизмах** | ТМ | Таблица: машина, марка, количество | ТТК + ПС |
| 8 | **Охрана труда, ПБ, экология** | ОТ | Правила по профессиям и видам работ | ТТК (виды работ) → шаблоны |
| 9 | **Лист ознакомления** | ЛО | Таблица: ФИО, профессия, подпись, дата | РК (профессии) |

### 3.2. Графическая часть (чертежи)

| № | Чертёж | Обозначение | Формат | Источник данных |
|---|--------|-------------|--------|-----------------|
| 1 | **Стройгенплан** | СГП | А1, М 1:500 | ОПР (зоны, дороги, склады) |
| 2 | **Технологические схемы** | ТС | А1, по этапам | ТТК (последовательность операций) |
| 3 | **Схемы строповки и складирования** | СС | А1 | ПС (грузы, стропы) |
| 4 | **Технические характеристики техники** | ТХ | А1 | ТМ (таблица на чертеже) |
| 5 | **Календарный график производства работ** | КГ | А1 | ТТК + РК + ПОС (диаграмма Ганта) |

### 3.3. Служебные элементы

- **Основная надпись (штамп ГОСТ 21.1101-2013)** — на каждом листе
- **Шифр документа** — формат `ГГГГ/НН-НН-ППР` (например, `2023/02-05-ППР`)
- **Листы изменений** — в штампе

## 4. Входные данные

### 4.1. Обязательные входы

```
┌─────────────┐     ┌─────────────┐
│     ПОС      │     │   ПД/РД     │
│ (Проект орг- │     │ (Проектная/ │
│  ции стр-ва) │     │  Рабочая    │
│              │     │  документа- │
│              │     │  ция)       │
└──────┬───────┘     └──────┬──────┘
       │                    │
       │  ┌─────────────┐  │
       │  │  Реквизиты   │  │
       │  │  (заказчик,  │  │
       │  │  подрядчик,  │  │
       │  │  объект,     │  │
       │  │  шифр и пр.) │  │
       │  └──────┬───────┘  │
       │         │          │
       ▼         ▼          ▼
   ┌────────────────────────────┐
   │    ppr_generator input     │
   └────────────────────────────┘
```

**ПОС предоставляет:**
- Сроки строительства (начало/конец по этапам)
- Общие решения по организации стройплощадки
- Ограничения (охранные зоны, существующие коммуникации)
- Потребность в основных ресурсах (укрупнённо)
- Календарный план (укрупнённый)

**ПД/РД предоставляет:**
- Состав и объёмы работ (по разделам)
- Спецификации материалов и конструкций
- Конструктивные решения (чертежи)
- Требования к качеству (допуски, марки, классы)
- Характеристики объекта (габариты, высотные отметки, привязки)

**Реквизиты (вводятся пользователем):**
- Наименование объекта
- Заказчик (наименование, реквизиты)
- Подрядчик (наименование, реквизиты)
- Шифр проекта
- Разработчик ППР (организация, ФИО, должности)
- Утверждающий, согласующие лица

### 4.2. Формат входных данных

ПОС и ПД/РД поступают в систему в виде:
- PDF (сканы или электронные документы)
- DOCX (текстовые разделы)
- DXF/DWG (чертежи)

Модуль `ppr_generator` не парсит эти форматы самостоятельно — он получает уже структурированные данные через интерфейс модуля-парсера (существующий `parser_engine` + `dxf_parser` из `is_generator`).

### 4.3. Pydantic-модели входных данных (предварительно)

```python
class PPRInput(BaseModel):
    """Входные данные для генерации ППР"""
    # Реквизиты
    object_name: str                        # Наименование объекта
    project_code: str                       # Шифр проекта (напр. "2023/02-05")
    customer: OrganizationInfo              # Заказчик
    contractor: OrganizationInfo             # Подрядчик
    developer: DeveloperInfo                # Разработчик ППР
    
    # Данные из ПОС
    construction_schedule: ScheduleData     # Сроки строительства
    site_constraints: list[Constraint]      # Ограничения стройплощадки
    pos_resource_needs: ResourceNeeds       # Потребность в ресурсах (из ПОС)
    
    # Данные из ПД/РД
    work_types: list[WorkTypeItem]          # Виды работ с объёмами
    material_specs: list[MaterialSpec]      # Спецификации материалов
    structural_solutions: list[StructRef]   # Ссылки на конструктивные решения
    quality_requirements: list[QualityReq]  # Требования к качеству
    
    # Параметры генерации
    include_graphics: bool = True           # Генерировать графическую часть
    output_format: Literal["pdf", "docx", "both"] = "both"
```

### 5.2. Оркестратор (ppr_generator.py)

Оркестратор управляет последовательностью генерации разделов и агрегацией данных между ними. Это ключевой класс модуля — он определяет pipeline:

```
PPRGenerator.generate(input: PPRInput) -> PPRResult
    │
    ├── 1. Анализ входных данных
    │   ├── Определение видов работ из РД
    │   ├── Выбор ТТК-генераторов через ttk_registry
    │   └── Планирование графики
    │
    ├── 2. Генерация разделов ПЗ (с зависимостями)
    │   │
    │   │  Уровень 0 (без зависимостей):
    │   ├── → general_data(input)
    │   ├── → work_organization(input)
    │   │
    │   │  Уровень 1 (зависят от Уровня 0):
    │   ├── → ttk_welding(input, work_org_data)
    │   ├── → ttk_sheet_pile(input, work_org_data)
    │   ├── → ttk_anticorrosion(input, work_org_data)
    │   ├── → ... (по видам работ)
    │   │
    │   │  Уровень 2 (зависят от ТТК):
    │   ├── → ps_works(ttk_results)
    │   ├── → quality_control(ttk_results)
    │   │
    │   │  Уровень 3 (агрегируют данные):
    │   ├── → manpower(ttk_results, schedule)
    │   ├── → machinery(ttk_results, ps_results)
    │   │
    │   │  Уровень 4 (зависят от всех предыдущих):
    │   ├── → safety(ttk_results, manpower_data, ps_results)
    │   ├── → attestation(manpower_data)
    │   │
    │   └── Результат: dict[section_id, SectionResult]
    │
    ├── 3. Генерация графической части
    │   ├── → site_plan(work_org_data)
    │   ├── → tech_schemes(ttk_results)
    │   ├── → slinging(ps_results)
    │   ├── → equipment_table(machinery_data)
    │   └── → schedule(ttk_results, manpower_data, construction_schedule)
    │
    ├── 4. Генерация служебных страниц
    │   ├── → title_page(input)
    │   └── → approval_sheet(input)
    │
    ├── 5. Сборка документа
    │   ├── → pdf_exporter.compile(all_results) → PDF
    │   └── → docx_exporter.compile(all_results) → DOCX
    │
    └── 6. Результат
        └── PPRResult(pdf_path, docx_path, sections_meta, stats)
```

### 5.3. Система ТТК — ядро модуля

ТТК (технологическая карта) — наиболее содержательная и сложная часть ППР. Каждая ТТК описывает один вид работ и имеет фиксированную структуру из 4 подразделов.

#### Базовый класс ТТК

```python
class BaseTTKGenerator(ABC):
    """Базовый класс генератора технологической карты"""
    
    @property
    @abstractmethod
    def work_type(self) -> str:
        """Вид работ (ключ из ttk_registry)"""
        ...
    
    @abstractmethod
    def generate_scope(self, input: PPRInput, org_data: OrgDataResult) -> TTKScope:
        """Область применения: где и когда применяется, объёмы"""
        ...
    
    @abstractmethod
    def generate_technology(self, input: PPRInput, scope: TTKScope) -> TTKTechnology:
        """Организация и технология выполнения работ"""
        ...
    
    @abstractmethod
    def generate_quality(self, input: PPRInput, technology: TTKTechnology) -> TTKQuality:
        """Требования к качеству и контролю"""
        ...
    
    @abstractmethod
    def generate_resources(self, input: PPRInput, technology: TTKTechnology) -> TTKResources:
        """Потребность в материально-технических ресурсах"""
        ...
    
    def generate(self, input: PPRInput, org_data: OrgDataResult) -> TTKResult:
        """Полная генерация ТТК (шаблонный метод)"""
        scope = self.generate_scope(input, org_data)
        technology = self.generate_technology(input, scope)
        quality = self.generate_quality(input, technology)
        resources = self.generate_resources(input, technology)
        return TTKResult(
            work_type=self.work_type,
            scope=scope,
            technology=technology,
            quality=quality,
            resources=resources
        )
```

#### Реестр ТТК

```python
class TTKRegistry:
    """Реестр: маппинг вид работ → генератор ТТК"""
    
    _generators: dict[str, type[BaseTTKGenerator]] = {}
    
    @classmethod
    def register(cls, work_type: str, generator_cls: type[BaseTTKGenerator]):
        cls._generators[work_type] = generator_cls
    
    @classmethod
    def get(cls, work_type: str) -> type[BaseTTKGenerator] | None:
        return cls._generators.get(work_type)
    
    @classmethod
    def select_for_project(cls, work_types: list[str]) -> list[BaseTTKGenerator]:
        """Выбрать генераторы ТТК для заданных видов работ"""
        generators = []
        for wt in work_types:
            gen_cls = cls._generators.get(wt)
            if gen_cls:
                generators.append(gen_cls())
            # Если генератора нет — использовать универсальный (LLM-генерация)
        return generators
```

#### Типы ТТК (первый релиз)

На основе анализа ППР и типовых видов работ в гидротехническом/промышленном строительстве:

| Ключ | Класс | Вид работ |
|------|-------|-----------|
| `welding` | `TTKWelding` | Сварные соединения металлоконструкций |
| `sheet_pile` | `TTKSheetPile` | Устройство шпунтового ограждения |
| `anticorrosion` | `TTKAnticorrosion` | Антикоррозийное покрытие |
| `concrete` | `TTKConcrete` | Бетонные работы |
| `earthwork` | `TTKEarthwork` | Земляные работы |
| `metalwork` | `TTKMetalwork` | Монтаж металлоконструкций |

В будущем реестр расширяется: `piling` (сваи), `waterproofing` (гидроизоляция), `roadwork` (дорожные работы), `pipelines` (трубопроводы) и т.д.

## 6. Поток данных между разделами

Разделы ППР не генерируются изолированно — данные перетекают между ними. Это принципиальное архитектурное решение: ТТК формирует данные, которые затем агрегируются в ресурсные разделы и раздел ОТ.

```
                        ПОС + ПД/РД
                            │
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
        general_data  work_org    ttk_registry
                                (выбор ТТК)
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
              ttk_welding    ttk_sheet_pile   ttk_anticorrosion
                    │               │               │
                    └───────┬───────┘───────────────┘
                            │
                    ┌───────┼───────┐
                    ▼       ▼       ▼
                ps_works  quality  ──┐
                                    │
                    ┌───────────────┘
                    ▼
              ┌─────┼─────┐
              ▼     ▼     ▼
          manpower  machinery  safety
              │     │
              ▼     ▼
         attestation  equipment_table
              │
              ▼
         schedule (КГ)
              │
              ▼
        ┌─────┼─────┐
        ▼     ▼     ▼
    site_plan  tech  slinging
              schemes
```

### Типы данных, передаваемых между разделами

| Откуда | Куда | Данные |
|--------|------|--------|
| ТТК | ps_works | Характеристики грузов, типы ПС, радиусы действия |
| ТТК | quality | Критерии приёмки, допуски, методы контроля |
| ТТК | manpower | Состав бригад по профессиям и разрядам |
| ТТК | machinery | Потребность в механизмах по видам работ |
| ТТК | safety | Виды работ → подбор шаблонов ОТ |
| ТТК | schedule | Трудоёмкость по видам работ |
| ps_works | safety | Типы ПС → правила для крановщиков, стропальщиков |
| manpower | safety | Профессии → правила ОТ |
| manpower | attestation | Список профессий для листа ознакомления |
| manpower | schedule | Количество рабочих → длительность операций |
| machinery | equipment_table | Перечень техники для чертежа |
| work_org | site_plan | Зоны стройплощадки, дороги, склады |
| work_org | safety | Решения по стройплощадке → требования ОТ |

## 7. Раздел ОТ, ПБ, экология — шаблонный подход

Раздел охраны труда — самый объёмный в ППР (23 страницы из 95 в примере). Он генерируется преимущественно по шаблонам, с подстановкой параметров конкретного объекта.

### Принцип

Для каждой профессии/вида работ существует типовой набор правил, который слабо зависит от конкретного объекта. Разница — в параметрах: высота, грузоподъёмность, тип ПС, габариты и т.д.

### Шаблон правил ОТ (пример: стропальщик)

```json
{
  "profession": "Стропальщик",
  "code": "rigger",
  "sections": [
    {
      "title": "Общие требования охраны труда",
      "template": "К выполнению работ по строповке грузов допускаются лица не моложе {min_age} лет, прошедшие медицинский осмотр, вводный инструктаж, первичный инструктаж на рабочем месте и имеющие удостоверение стропальщика.",
      "params": {
        "min_age": 18
      }
    },
    {
      "title": "Требования охраны труда перед началом работы",
      "items": [
        "Осмотреть стропы, траверсы, захваты и другие съемные грузозахватные приспособления",
        "Проверить наличие и исправность средств индивидуальной защиты",
        "Получить задание от лица, ответственного за безопасное производство работ с применением ПС грузоподъёмностью до {crane_capacity} т"
      ],
      "params": {
        "crane_capacity": null
      }
    }
  ]
}
```

Параметры со значением `null` заполняются из данных проекта (мощность крана, высота здания и т.д.).

## 8. Графическая часть

### 8.1. Стройгенплан (СГП)

**Сложность:** высокая. Требует пространственного размещения зон на плане площадки.

**Подход — двухуровневый:**
1. **Базовый уровень (автоматический):** генерация схемы с упрощённым размещением зон (склады, дороги, бытовки, опасные зоны) по алгоритму на основе данных ПОС
2. **Продвинутый уровень (ручная корректировка):** экспорт в DXF для доработки в AutoCAD

**Формат вывода:** SVG → PDF (векторный) + DXF (для AutoCAD)

### 8.2. Технологические схемы

**Подход:** пошаговые схемы производства работ. Генерируются на основе данных из ТТК (последовательность операций).

**Формат:** SVG → PDF, по 1–4 листа на каждый вид работ.

### 8.3. Схемы строповки

**Подход:** библиотека типовых схем строповки для стандартных грузов (шпунт, балка, плита, секция). Выбор схем — по типам грузов из РД.

**Формат:** SVG → PDF, 1 лист.

### 8.4. Технические характеристики техники

**Подход:** таблица с характеристиками на чертежном листе. Данные — из раздела «Потребность в технике».

**Формат:** SVG → PDF, 1 лист.

### 8.5. Календарный график (диаграмма Ганта)

**Подход:** линейный график производства работ. Данные — из ТТК (трудоёмкость) + ПОС (сроки) + manpower (количество рабочих).

**Формат:** SVG → PDF (векторная диаграмма Ганта), дополнительно — Excel (.xlsx) для редактирования.

## 11. Взаимодействие с is_generator и shared-модулями

### 11.1. Что НЕ переносится в shared

GOST-штамп в `is_generator` и основной надписи в `ppr_generator` — это **разные формы**:
- ИС: штамп по ГОСТ для исполнительных схем (форма 3, 4)
- ППР: штамп по ГОСТ 21.1101-2013 для проектной документации (форма 5, 6, 7)

Поэтому `gost_stamp.py` остаётся в `is_generator`, а `ppr_generator` получает свой собственный генератор основной надписи.

### 11.2. Что выделяется в shared

| Модуль | Что | Причина |
|--------|-----|---------|
| `shared/event_bus.py` | Базовый класс событийной системы | Общая логика для is_generator и ppr_generator |
| `shared/pdf_base.py` | Базовый класс PDF-экспорта (страницы, шрифты, отступы) | Общая логика компоновки PDF |

## 12. Отличия от типовых ППР-генераторов на рынке

| Аспект | Типовые решения | ppr_generator (АСД) |
|--------|-----------------|---------------------|
| **Источник данных** | Ручной ввод | Автоматический парсинг ПОС + ПД/РД |
| **ТТК** | Библиотека типовых ТТК (копипаст) | Генерация с адаптацией под конкретные объёмы и условия |
| **Календарный план** | Ручное построение | Автоматический на основе трудоёмкости из ТТК + сроков из ПОС |

| **Скорость** | 2–5 дней на один ППР | Целевая: 30–60 минут |

## 14. Открытые вопросы

1. **Универсальный ТТК-генератор (fallback):** что делать, если вид работ не зарегистрирован в ttk_registry? Варианты: (A) генерировать через LLM «на лету», (B) выдать предупреждение и пропустить, (C) использовать универсальный шаблон с минимальным заполнением.

2. **Стройгенплан — уровень автоматизации:** полная автогенерация (AI расставляет зоны) крайне сложна и ненадёжна. Более реалистичный подход — генерация упрощённой схемы (прямоугольники зон с подписями) для предварительного просмотра, + экспорт DXF для ручной доработки в AutoCAD.

3. **LLM-участие:** на каком этапе привлекать LLM (Gemma 4 31B)? Для текстовых описаний технологии — да. Для табличных данных (ресурсы, кадры, техника) — нет, только вычисления. Для ОТ — шаблоны + LLM-дополнение?

4. **Календарный график — редактируемость:** генерировать только PDF или также Excel для ручной корректировки? Рекомендация: оба формата.

5. **Формат чертежей:** только PDF (для печати) или также DXF (для AutoCAD)? Рекомендация: оба, DXF начиная с Этапа 3.

6. **Верстка ПЗ:** DOCX с последующим преобразованием в PDF, или прямая генерация PDF (ReportLab/Playwright)? DOCX даёт пользователю возможность редактирования, PDF — более качественную верстку. Рекомендация: DOCX как основной формат ПЗ, PDF — финальная сборка всего комплекта.

│   ├── Юрист

│   │   │   ├── journal/       ← Реконструкция журнала
│   │   │   ├── incoming_control/ ← АВК / ЖВК

### 15.2. Ключевые связи ppr_generator ↔ id_pipeline

| Связь | Направление | Описание |
|-------|------------|----------|
| TTK Registry → AOSRChainBuilder | ppr → id_pipeline | ТТК определяет технологическую последовательность для цепочек АОСР |
| ТТК quality → AVKGenerator | ppr → id_pipeline | Критерии входного контроля из ТТК используются для генерации АВК |
| ТТК resources → KS2Builder | ppr → id_pipeline | Состав бригад и техника — для обоснования объёмов в КС-2 |
| InventoryEngine → ppr_generator | id_pipeline → ppr | Лакуна `doc_type="ppr"` инициирует генерацию ППР |
| DocumentLifecycleManager → ppr | id_pipeline → ppr | Управление жизненным циклом сгенерированного ППР (DRAFT → REVIEW → APPROVED) |
| ChronoValidator → ppr | id_pipeline → ppr | Проверка хронологической совместимости дат в ППР с остальным комплектом ИД |

### 15.3. Стратегический контекст

АСД — это не набор генераторов, а система восстановления полного комплекта ИД на проблемных ОКС. Стратегическая цель: **подрядчик получает деньги за выполненные СМР**. ППР — важный элемент комплекта, но не блокер оплаты. Приоритеты конвейера:

1. **P0:** АОСР (якорный документ) + АВК (без них сертификаты не принимаются)
2. **P1:** Журнал работ + Лаб. протоколы (часто утрачиваются или отсутствуют)
3. **P2:** ППР (для комплектности, не блокирует оплату) + КС-2/КС-3 (агрегация из АОСР)
4. **P3:** Цифровая ИД (xml, ГОСТ Р 70108-2025 — на будущее)

Подробная архитектура конвейера — в `docs/id_pipeline_architecture.md`.
