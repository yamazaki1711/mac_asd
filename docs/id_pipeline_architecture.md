## 1. Контекст и мотивация

АСД — это не набор независимых генераторов документов. Это система, которая заходит на проблемный ОКС и **выводит его к полному комплекту ИД, готовому к сдаче заказчику**. Полный комплект по Приказу 344/пр — это не один документ и не два. Это 13+ типов документов, которые связаны друг с другом хронологически, логически и нормативно.

Сегодня в коде есть три генератора, работающих изолированно:

| Модуль | Что генерирует | Статус |
|--------|---------------|--------|
| `is_generator` | Исполнительные схемы (ИС) — векторные чертежи с отклонениями | ✅ Работает (16 файлов, 99+ тестов) |
| `ppr_generator` | Проект производства работ (ППР) — комплект ПЗ + графика | 🔧 MVP (20 тестов) |
| `output_pipeline` | АОСР по Приказу 344/пр | 🔧 Только АОСР |

Проблема: эти модули не знают друг о друге. АОСР генерируется без учёта технологической последовательности из ТТК. ППР генерируется без связи с исполнительной документацией. Исполнительные схемы не знают, какие работы освидетельствованы в АОСР. А в реальности **все эти документы — части одного комплекта**, и они должны быть согласованы между собой.

## 2. Стратегическая цель в операционных терминах

```
Проблемный ОКС → Команда (4 чел. + Mac Studio + сканер) → Полный комплект ИД → Оплата
```

Система должна обеспечивать **все этапы** этого процесса:

| Этап | Что делает команда | Что делает АСД | Длительность |
|------|-------------------|----------------|-------------|
| **Фидинг** | Сканируют бумаги, копируют файлы на SSD | OCR + классификация в реальном времени | 1–3 дня |
| **Инвентаризация** | — | Анализ всего массива → отчёт «Что есть / Чего нет» | 1–2 дня |
| **Восстановление** | Согласуют, подтверждают, принимают решения | Генерация недостающих документов, проверка хронологии, связность комплекта | 5–15 дней |
| **Согласование** | Отправляют партии ИД стройконтролю | Формирование пакетов, отслеживание статуса | Параллельно |
| **Финализация** | Печать, подшивка, сдача | Сборка финального комплекта, нумерация, форматирование | 1–2 дня |

## 3. Архитектурные принципы

### Принцип 1: Двунаправленный конвейер

Нормальный поток данных при строительстве: от первоисточника (журнал) ко всем производным документам. Но при восстановлении ИД поток разворачивается: от сохранившихся фрагментов к реконструкции утраченного первоисточника.

```
Прямой ход (нормальное строительство):
  Журнал → АОСР → КС-2 → КС-3 → Оплата

Обратный ход (восстановление ИД):
  КС-2 + сертификаты + ТТН + опросы рабочих → Журнал → АОСР
```

Оба направления должны быть реализованы явно и одновременно. Система не знает заранее, какой путь потребуется для конкретного документа — она определяет это по результатам инвентаризации.

**Архитектурное следствие:** каждый генератор документа должен уметь работать в двух режимах:
- **Прямой** — на основе данных из «вышестоящего» документа (журнал → АОСР)
- **Обратный** — на основе данных из «нижестоящих» документов + данных от оператора (КС-2 + сертификаты → журнал)

### Принцип 2: Цепочечная генерация

Исполнительные документы не генерируются поштучно. Они образуют **цепочки** по технологическим картам (ТТК), где каждый документ ссылается на предыдущий и следующий этап работ.

```
ТТК (шпунт): забивка → срезка → антикоррозия → бетонирование ростверка

АОСР-1: забивка шпунта         → следующий этап: срезка
АОСР-2: срезка шпунта          → следующий этап: антикоррозия
АОСР-3: антикоррозия           → следующий этап: бетонирование ростверка
АОСР-4: бетонирование ростверка → следующий этап: —
```

**Архитектурное следствие:** генерация АОСР работает не по одному акту, а **цепочками** — вся последовательность по одной ТТК за раз, с автоматической связкой «предыдущий → следующий этап». Текущий `output_pipeline` генерирует отдельные АОСР — это верно для forensic-режима (найти и исправить один акт), но для генерации с нуля нужен режим `generate_chain(work_type, date_range)`.

### Принцип 3: Система предлагает — человек решает

Каждый сгенерированный документ проходит через жизненный цикл с чёткими состояниями:

```
DRAFT → REVIEW → APPROVED → SUBMITTED → ACCEPTED → PRINTED
                                    ↓
                                 REJECTED → (цикл исправлений)
```

- **DRAFT** — система сгенерировала документ автоматически
- **REVIEW** — оператор видит документ, его исходные данные, предупреждения о нестыковках
- **APPROVED** — оператор подтвердил корректность
- **SUBMITTED** — документ отправлен стройконтролю заказчика
- **ACCEPTED** — стройконтроль принял документ
- **REJECTED** — стройконтроль вернул с замечаниями → цикл исправлений

**Система никогда не переводит документ в APPROVED автоматически.** Это фундаментальное правило: АСД — штурман, а не автопилот. Все критические решения принимает человек.

### Принцип 4: Хронологическая совместимость

Все документы в комплекте ИД должны быть хронологически согласованы. Это не просто «дата A раньше даты B» — это система правил:

```
Дата поставки (ТТН) ≤ Дата входного контроля ≤ Дата использования (АОСР)
Дата испытания образца ≤ Дата использования материала
Дата начала работы (Журнал) ≤ Дата АОСР ≤ Дата КС-2
Дата АОСР_i ≤ Дата АОСР_(i+1)  (в рамках одной ТТК)
```

**Архитектурное следствие:** в системе должен быть **хронологический валидатор** — модуль, который проверяет весь комплект ИД на хронологическую совместимость при любом изменении. Это не одноразовая проверка, а непрерывный инвариант, который поддерживается на протяжении всего процесса восстановления.

### Принцип 5: Партионная отправка

Полный комплект ИД не сдаётся единовременно. Документы группируются в **партии** по видам работ, технологическим этапам или периодам строительства. Каждая партия проходит свой цикл согласования независимо.

```
Партия 1: Земляные работы (АОСР + Журнал + ИС + сертификаты)
Партия 2: Фундаменты (АОСР + Журнал + ИС + лаб. протоколы)
Партия 3: Металлоконструкции (АОСР + Журнал + ИС + сертификаты + УЗК)
...
```

**Архитектурное следствие:** система должна уметь формировать партии, отслеживать статус каждой партии и каждого документа внутри партии, и не допускать отправки партии с незакрытыми хронологическими зависимостями от документов из других партий.

## 6. Ключевые подсистемы — детальная спецификация

### 6.1. Inventory Engine (Инвентаризация)

Инвентаризация — первый шаг после фидинга. Система анализирует весь собранный массив документов и определяет, что есть, чего нет и в каком состоянии.

**Вход:** результаты Ingestion Pipeline (OCR → Classify → Extract → KAG)

**Выход:** `InventoryReport`

```python
class InventoryReport(BaseModel):
    """Отчёт инвентаризации: Что есть / Чего нет"""
    project_code: str
    object_name: str

    # Что найдено
    found: List[FoundDocument]       # Найденные документы с атрибутами

    # Чего нет
    gaps: List[DocumentGap]          # Лакуны: документы, которые должны быть, но отсутствуют

    # Что повреждено
    damaged: List[DamagedDocument]   # Документы с проблемами (нечитаемые, без дат, etc.)

    # Статистика
    total_expected: int              # Ожидаемое количество документов (по 344/пр + ТТК)
    total_found: int                 # Найдено
    completeness_pct: float          # Полнота комплекта (%)
```

```python
class DocumentGap(BaseModel):
    """Лакуна — отсутствующий документ"""
    doc_type: str                    # Тип документа (АОСР, Журнал, АВК, ...)
    work_type: str                   # Вид работ (welding, concrete, ...)
    reason: GapReason                # Причина отсутствия
    recoverability: Recoverability   # Восстановимость
    required_data: List[str]         # Какие данные нужны для генерации
    available_data: List[str]        # Какие данные уже есть (косвенные)

class GapReason(str, Enum):
    NEVER_CREATED = "never_created"       # Никогда не создавался
    LOST = "lost"                         # Утрачен
    REJECTED = "rejected"                 # Забракован стройконтролем
    INCOMPLETE = "incomplete"             # Создан, но не заполнен

class Recoverability(str, Enum):
    AUTO_GENERATABLE = "auto"             # Можно сгенерировать автоматически
    NEEDS_OPERATOR_DATA = "needs_data"    # Нужны данные от оператора
    NEEDS_FIELD_WORK = "needs_field"      # Нужно обследование на объекте
    NEEDS_LAB_WORK = "needs_lab"          # Нужны лабораторные испытания
    NOT_RECOVERABLE = "not_recoverable"   # Невосстановим (редкий случай)
```

### 6.2. Journal Reconstructor (Обратный ход) ✅ Реализован (Package 11)

Самый сложный сценарий: журнал работ утрачен или не заполнен за период. Система реконструирует его из косвенных данных.

> **Детали реализации:** см. `docs/CORE_LOGIC_DESIGN.md` §9 — Journal Reconstructor v2 с интеграцией Evidence Graph v2 и HITL System.

**Прямой ход** (журнал есть):
```
Журнал → данные о датах, объёмах, бригадах → АОСР, КС-2
```

**Обратный ход** (журнала нет):
```
КС-2 (объёмы по периодам)
+ Сертификаты (даты поставки, объёмы)

Ключевой момент: **опросы рабочих — данные, которые система не может получить сама**. Система формулирует вопросы, но ответы даёт человек. Это чистый human-in-the-loop:

```

### 6.3. Chain Builder (Цепочечная генерация АОСР) ✅ Реализован (Package 11)

АОСР генерируются не поштучно, а цепочками по ТТК — с автоматической связкой «предыдущий этап → следующий этап».

> **Детали реализации:** см. `docs/CORE_LOGIC_DESIGN.md` — Chain Builder с поддержкой цепочечной генерации и валидации через Evidence Graph v2.

```python
class AOSRChainBuilder:
    """Сборщик цепочек АОСР по технологическим картам"""

    def build_chain(
        self,
        ttk_result: TTKResult,
        date_range: DateRange,
        project_data: ProjectData,
        existing_aosr: List[AOSRRecord] = None,
    ) -> List[AOSRRecord]:
        """
        Собрать цепочку АОСР для одного вида работ.

        Алгоритм:
        1. Извлечь последовательность операций из ТТК
        2. Разбить операции на освидетельствуемые этапы
        3. Для каждого этапа:
           a. Определить дату (из журнала или оценить по хронологии)
           b. Определить объёмы (из ВОР или КС-2)
           c. Заполнить «следующий этап» из следующего АОСР в цепочке
           d. Заполнить «предыдущий этап» из предыдущего АОСР
        4. Проверить связность цепочки
        5. Проверить хронологическую совместимость
        """
        ...

    def _resolve_next_stage(
        self,
        current_operation: TTKOperation,
        operations: List[TTKOperation],
    ) -> str:
        """Определить «следующий этап» для текущего АОСР"""
        idx = operations.index(current_operation)
        if idx + 1 < len(operations):
            return operations[idx + 1].name
        return "—"
```

Параллельные процессы: на объекте могут идти одновременно несколько ТТК. Например, на одной захватке идёт бетонирование, а на другой — монтаж металлоконструкций. Цепочки строятся **независимо**, но хронологический валидатор проверяет, что даты не противоречат друг другу (нельзя монтировать конструкцию до того, как готов фундамент).

## 8. Позиционирование существующих модулей

### 8.1. is_generator — внутри конвейера

`is_generator` становится одним из генераторов внутри `id_pipeline/generators/`. Он не переписывается — его API вызывается оркестратором при необходимости сгенерировать исполнительную схему.

**Изменения минимальны:**
- `is_generator` получает интерфейс для приёма данных из KAG (вместо отдельного DXF-файла)
- Результат ИС связывается с АОСР через KAG
- Жизненный цикл ИС управляется через DocumentLifecycleManager

### 8.2. ppr_generator — внутри конвейера

`ppr_generator` также становится одним из генераторов. ППР — это один из 13+ типов документов, которые могут потребоваться для полного комплекта. Его генерация инициируется Inventory Engine, если обнаружена лакуна типа `doc_type="ppr"`.

**Изменения минимальны:**
- ППР-генерация инициируется через общий интерфейс конвейера
- Результат ППР связывается с остальными документами через KAG
- ТТК Registry из `ppr_generator` используется `ChainBuilder` для построения цепочек АОСР

### 8.3. output_pipeline — интегрируется в Chain Builder

Текущий `output_pipeline` (генерация АОСР по 344/пр) становится частью `AOSRChainBuilder`. Его функциональность по генерации отдельных АОСР сохраняется для forensic-режима (найти и исправить один акт), а цепочечный режим добавляется поверх.

## 10. Чёткая граница: система не делает

АСД — мощный инструмент, но у него есть границы. Следующие действия система **не выполняет** и **не должна выполнять**:

1. **Не звонит в лабораторию** — это делает оператор. Система может подготовить письмо-обращение, но не отправляет его.
2. **Не принимает решение о подлоге** — система генерирует корректные по форме документы, но решение об их юридическом статусе принимает человек.
3. **Не переводит документы в APPROVED автоматически** — каждый DRAFT проходит через REVIEW оператором.
4. **Не фиксирует в логах ничего компрометирующего** — система не создаёт записей вида «сгенерирован поддельный протокол от [дата]». Все документы — «сгенерированы на основе доступных данных».
5. **Не заменяет профессиональное суждение** — система строит логические цепочки, выявляет нестыковки и предлагает варианты. Окончательное решение — за инженером, юристом, руководителем.
