# АСД — АРХИТЕКТУРНО-СТРОИТЕЛЬНАЯ СИСТЕМА

## КОНЦЕПЦИЯ v11.0

**Дата:** 17 апреля 2026
**Статус:** Активная разработка (Package 1 завершен, Package 2-10 в очереди)
**Целевая платформа:** Mac Studio, Apple Silicon M4 Max, 128GB Unified Memory
**LLM:** Llama 3.3 70B, Gemma 4 31B/9B, Qwen3 32B (локально, MLX через LLMEngine)

**Философия:** Полностью локальная, оффлайн, автономная

## 1. ЧТО ТАКОЕ АСД

АСД — единая платформа ИИ-модулей для автоматизации документооборота строительной компании.

Не заменяет специалистов. **Усиливает** их.

### Проблема

Строительная компания тонет в документах:

- **Договоры** — сотни страниц, ловушки, протоколы разногласий
- **ПД/РД** — проектная и рабочая документация, коллизии, неучтённые объёмы
- **ВОР** — ведомости объёмов работ, сверки, расхождения
- **Сметы** — ЛСР, ГЭСН/ФЕР, допсоглашения
- **ИД** — АОСР, акты входного контроля, журналы работ
- **Корреспонденция** — заявки, уведомления, письма, отправка Заказчику

Всё проверяется, сверяется и генерируется **вручную**. Ошибки стоят денег и времени.

### Что автоматизирует АСД

| Модуль | Задачи |
|--------|--------|
| **Юрист** | Экспертиза → протокол разногласий → претензия → иск в арбитраж |
| **ПТО** | Входной контроль ПД → анализ → сверка ВОР → акты (АОСР и др.) |
| **Сметчик** | Сверка ВОР со сметой → создание ЛСР → осмечивание допсоглашений |
| **Делопроизводитель** | Заявки, уведомления, корреспонденция, отправка документов Заказчику |

### Что АСД НЕ делает

- Не заменяет юриста, ПТО-инженера, сметчика, делопроизводителя
- Не принимает решений — предоставляет информацию и рекомендации
- Не подписывает, не утверждает документы
- Не является сметной программой — для составления ЛСР используется **Smeta.Cloud** (cloud.smeta.ru)

## 2. КЛЮЧЕВЫЕ ПРИНЦИПЫ

### 2.1. Полная локальность

Все данные остаются на Mac Studio. Никаких облаков, API-ключей, зависимостей от интернета.

Единственный профиль — `mac_studio` (MLX на Mac Studio M4 Max 128GB). Все модели работают через нативный MLX-интерфейс.

**Почему:** коммерческая тайна, госзаказчики (ФКУ «Ространсмодернизация»), юр. конфиденциальность, работа без стабильного интернета.

### 2.2. Модульность

7 агентов — самостоятельные единицы с чётким интерфейсом через MCP. Используют общую инфраструктуру (LLM, embeddings, БД, поиск), но не зависят друг от друга.

### 2.3. Компаундинг знаний

Система умнеет с каждым документом. Загруженный договор → нормативные ссылки → граф связей. Сверенная ВОР → объёмы записаны → будущие сверки точнее.

### 2.4. Специалист в контуре

Человек **всегда** принимает финальное решение. АСД подсвечивает проблемы, предлагает варианты, готовит черновики.

### 2.5. LangGraph + LLMEngine как ядро оркестрации

Система построена на LangGraph StateGraph — каждый из 7 агентов является узлом графа. Hermes (Llama 3.3 70B) — оркестратор, маршрутизирующий задачи. LLMEngine обеспечивает единый интерфейс к LLM через MLX.

## 3. МОДУЛИ И MCP ИНСТРУМЕНТЫ

### 3.1. Юрист (6 инструментов)

Полный юридический цикл: экспертиза → протокол → претензия → арбитраж.

| # | Инструмент | Описание |
|---|-----------|----------|
| 1 | `asd_upload_document` | Загрузка + парсинг документа |
| 2 | `asd_analyze_contract` | Юридическая экспертиза (БЛС + Map-Reduce + LightRAG) |
| 3 | `asd_normative_search` | Поиск по нормативной базе (Graph+Vector) |
| 4 | `asd_generate_protocol` | Протокол разногласий (DOCX) |
| 5 | `asd_generate_claim` | Претензия при неоплате СМР (DOCX) |
| 6 | `asd_generate_lawsuit` | Исковое заявление в арбитраж (DOCX) |

**Workflow:**

```
Договор → экспертиза → ловушки найдены → протокол разногласий
  ↓
СМР завершён, нет оплаты → претензия
  ↓
Претензия не удовлетворена → исковое заявление в арбитраж
```

### 3.2. ПТО-инженер (4 инструмента)

Входной контроль ПД → комплексный анализ → сверка ВОР → генерация актов.

| # | Инструмент | Описание |
|---|-----------|----------|
| 7 | `asd_vor_check` | Сверка ВОР с ПД (объёмы, единицы, наименования) |
| 8 | `asd_pd_analysis` | Комплексный анализ ПД (коллизии, неучтённые объёмы/материалы) |
| 9 | `asd_generate_act` | Генерация акта (АОСР, входной контроль, скрытые работы) |
| 10 | `asd_id_completeness` | Проверка комплектности ИД |

**Workflow:**

```
ПД получена → входной контроль → комплексный анализ → коллизии, неучтённые
  ↓
ВОР получен → сверка с ПД → расхождения
  ↓
Строительство → АОСР, акты входного контроля, журнал работ
  ↓
СМР завершены → комплектация ИД
```

### 3.3. Инженер-сметчик (3 инструмента)

Сверка ВОР со сметой → создание ЛСР → осмечивание допсоглашений.

| # | Инструмент | Описание |
|---|-----------|----------|
| 11 | `asd_estimate_compare` | Сверка ВОР со сметным расчётом |
| 12 | `asd_create_lsr` | Создание ЛСР по ВОРу (если сметы нет) |
| 13 | `asd_supplement_estimate` | Осмечивание допсоглашения |

**Workflow:**

```
ВОР проверен ПТО
  ↓
Есть смета в ПД? → Да → сверка ВОР со сметой
  ↓
  Нет → создание ЛСР по ВОРу (ГЭСН/ФЕР из открытых источников)
  ↓
Допсоглашение → осмечивание новых объёмов → фин. влияние
```

### 3.4. Делопроизводитель (4 инструмента)

Регистрация → генерация писем → отправка Заказчику → отслеживание сроков.

| # | Инструмент | Описание |
|---|-----------|----------|
| 14 | `asd_register_document` | Регистрация входящего документа |
| 15 | `asd_generate_letter` | Генерация письма/уведомления/заявки |
| 16 | `asd_prepare_shipment` | Подготовка отправки Заказчику (сопроводительное + реестр) |
| 17 | `asd_track_deadlines` | Отслеживание сроков ответа |

**Workflow:**

```
Входящий документ → регистрация → классификация → дедлайн
  ↓
Нужно письмо → генерация по шаблону → реквизиты из договора
  ↓
Отправка ИД Заказчику → адрес из договора → сопроводительное → реестр
  ↓
Отслеживание → напоминание о дедлайнах
```

## 4. ЖИЗНЕННЫЙ ЦИКЛ ПРОЕКТА В АСД

```
1. Получение ПД от Заказчика
   → ПТО: входной контроль + комплексный анализ
   → Делопроизводитель: регистрация

2. Получение ВОР
   → ПТО: сверка ВОР с ПД
   → Сметчик: сверка со сметой или создание ЛСР

3. Заключение договора
   → Юрист: экспертиза + протокол разногласий
   → Делопроизводитель: регистрация, отправка

4. Строительство
   → ПТО: генерация АОСР, актов, журналов
   → Делопроизводитель: уведомления

5. Дополнительные объёмы (допсоглашение)
   → ПТО: анализ нового ВОР
   → Сметчик: осмечивание
   → Юрист: экспертиза допсоглашения

6. СМР завершены
   → ПТО: комплектация ИД
   → Делопроизводитель: отправка Заказчику

7. Нет оплаты
   → Юрист: претензия → Делопроизводитель: отправка

8. Претензия не удовлетворена
   → Юрист: исковое заявление в арбитраж
```

## 5. СМЕНА ПАРАДИГМЫ: v10.0 → v11.0

### Было (v10.0 — внешняя LLM, Linux)

```
Пользователь → Web UI → FastAPI (46 endpoints) → LLM Engine → OpenRouter API
                         ↓
                    MCP Server → Hermes
```

Толстый FastAPI-бэкенд, LLM Engine как прокси к облаку, MCP — тонкий слой.

### Стало (v11.0 — 7 агентов, LangGraph, Mac Studio)

```
Пользователь → Hermes (Llama 3.3 70B)
                       ↓ LangGraph StateGraph
             7 Agent Nodes (ПТО, Сметчик, Юрист, Закупщик, Логист, Делопроизводитель)
                       ↓
             LLMEngine (MLX)
                       ↓
             ParserEngine, RAG (Vector+Graph), БЛС, DOCX, PostgreSQL
```

**Hermes (Llama 3.3 70B)** = оркестратор (понимает intent, контекст, маршрутизирует)
**LangGraph StateGraph** = граф выполнения, каждый агент — узел
**LLMEngine** = единый интерфейс к LLM (MLX)
**FastAPI** = опционален (минимальный Web UI для файловых операций)

## 6. ТЕХНОЛОГИЧЕСКИЙ СТЕК

### 6.1. Apple Silicon: замена CUDA на MLX/Metal

На NVIDIA/CUDA ML-стек строился на CUDA + cuDNN. На Apple Silicon — принципиально другая архитектура:

| Задача | NVIDIA (CUDA) | Apple Silicon | Обоснование |
|--------|--------------|---------------|-------------|
| **GPU Compute** | CUDA | **Metal Performance Shaders (MPS)** | MPS — аналог CUDA для Apple GPU |
| **ML Framework** | PyTorch + CUDA | **PyTorch + MPS** / **MLX** | MLX — нативный фреймворк Apple для ML |
| **LLM Inference** | vLLM, TGI | **MLX** | Нативный фреймворк Apple для ML |
| **Embeddings** | sentence-transformers (CUDA) | **MLX Embeddings** | Без CUDA, через MPS/MLX |
| **Tensor Ops** | cuBLAS, cuDNN | **Metal BLAS**, **Accelerate** | Встроенные библиотеки Apple |
| **Memory** | GPU VRAM (отдельная) | **Unified Memory** (CPU+GPU общая) | Zero-copy, нет передачи между CPU↔GPU |

### 6.2. MLX — нативный путь Apple Silicon

Все модели в АСД запускаются через **MLX** — нативный фреймворк Apple для машинного обучения.

```
MLX (Apple)
  ├── Нативный фреймворк для Apple Silicon
  ├── Графовая оптимизация вычислений
  ├── Lazy evaluation
  ├── Поддержка Neural Engine
  └── Python API (mlx.core)
```

**Преимущества MLX:**

- Максимальная производительность на Apple Silicon
- Доступ к Neural Engine
- Полный контроль над inference
- Lazy evaluation — оптимизация графа вычислений
- Нативная интеграция с Metal Performance Shaders
- Единая кодовая база без внешних зависимостей

**Распределение моделей по MLX:**

| Компонент | Технология | Почему |
|-----------|-----------|--------|
| **Hermes (Llama 3.3 70B)** | MLX (4-bit, ~38GB) | Оркестратор, маршрутизация, понимание intent |
| **ПТО (Qwen2.5-VL-72B)** | MLX (4-bit) | Vision-анализ чертежей, сверка ВОР |
| **Сметчик, Юрист, Закупщик, Логист (Qwen3 32B)** | MLX (4-bit) | Расчёты, экспертиза, поиск, сравнение |
| **Делопроизводитель (Gemma 4 9B)** | MLX (4-bit) | Лёгкая модель, быстрая генерация писем |
| **Embeddings (bge-m3)** | MLX | Нативный inference |
| **Vision (Qwen2.5-VL-72B)** | MLX | Vision + OCR |
| **Кастомные ML-задачи** | MLX | Классификация, извлечение сущностей |

### 7.4. Thinking Mode

Gemma 4 поддерживает thinking mode — критически важно для:

- **Анализа договоров** — пошаговое выявление ловушек
- **Сверки ВОР/ПД** — детальное построчное сравнение
- **Генерации документов** — структурированное заполнение полей
- **Претензий и исков** — логическая последовательность, ссылки на законы

Thinking mode **отключается** для простых задач (классификация, статус) ради скорости.

### 8.2. Поток данных: полный цикл договора

```
Пользователь: "Проверь договор /Users/oleg/Docs/contract_РТМ-066.pdf"

1. Hermes (Gemma 4 31B):
   → Intent: анализ договора → Инструмент: asd_analyze_contract

2. MCP Server:
   a. Parser: PDF → текст (PyMuPDF или pytesseract/Qwen2.5-VL vision)
   b. Chunking: 150 страниц → ~40 чанков (4000 tokens, overlap 200)
   c. Map-Reduce: каждый чанк → Gemma 4 31B (thinking mode)
   d. БЛС: 27 ловушек → SQL проверка
   e. LightRAG: нормативный поиск (pgvector + NetworkX + RRF)
   f. CrossChecker: межсекционный анализ
   g. Результат: JSON (ловушки, статьи, рекомендации)

3. Hermes: "Найдено 12 ловушек. Сгенерировать протокол разногласий?"

4. Пользователь: "Да"
   Hermes → asd_generate_protocol(document_id=42)
   MCP    → DOCX протокол разногласий
   Hermes → "Протокол готов. Проверьте и отправьте Заказчику."

5. Пользователь: "Зарегистрируй и подготовь к отправке"
   Hermes → asd_register_document(file="protocol.docx", source="наш")
   Hermes → asd_prepare_shipment(documents=[protocol], contract="РТМ-066/22")
   MCP    → Сопроводительное письмо + реестр (адрес из договора)
   Hermes → "Пакет готов: протокол, сопроводительное, реестр"
```

## 9. БЛС — БАЗА ЛОВУШЕК СУБПОДРЯДЧИКА

27 ловушек в договорах подряда. Каждая — паттерн + рекомендация + ссылка на закон.

**Примеры:**

- «Заказчик вправе изменить объём работ без согласования цены» → ст. 743 ГК РФ
- «Штраф за просрочку 0.5% в день без ограничения» → ст. 333 ГК РФ
- «Приёмка работ только после полной оплаты» → ст. 711 ГК РФ

Хранится в PostgreSQL, загружается из YAML. Расширяется при обнаружении новых ловушек.

## 10. LIGHT RAG — ЕДИНЫЙ ПОИСК

```
Запрос → bge-m3 embeddings
       → Vector Search (pgvector, cosine similarity)
       → Graph Search (NetworkX, обход связей норм)
       → RRF Fusion (Reciprocal Rank Fusion, k=60)
       → Ранжированные результаты
```

Векторный поиск находит «похожее», графовый находит «связанное». Вместе — релевантное + контекстуальное.

## 12. ЭКОНОМИКА: "ЦЕНА ЗА ОБЪЕКТ"

Экономическая эффективность АСД строится на огромном объёме проектной и договорной документации (тысячи страниц на один объект), которую нужно анализировать по несколько раз.

### 12.1. Математика одного объекта
*   **Объём:** Типовой комплект ПД/РД + Договоры ≈ **2,000 страниц**.
*   **Токены:** ~1,000 токенов на страницу (текст + таблицы) = **2 млн токенов (сырой текст)**.
*   **Обработка:** Комплексный анализ (RAG + Map-Reduce + Reasoning) требует ≈ 3-кратного объёма токенов (на вход и выход) ≈ **6 млн токенов на один полный цикл анализа объекта**.

### 12.2. Сравнение стоимости

| Параметр | Облако (Claude 3.5 Sonnet) | Облако (Claude 3 Opus) | Mac Studio (Локально) |
| :--- | :--- | :--- | :--- |
| **Стоимость 1 объекта** | ~$50 - 100 | **~$600 - 1,000** | **$0** (включено) |
| **При 10 объектах/год** | ~$1,000 | **~$10,000** | **~$4,000** (CAPEX) |
| **При 50 объектах/год** | ~$5,000 | **~$50,000** | **~$4,000** (CAPEX) |

### 12.3. Режимы работы: "Охотник" vs "Хранитель"
Размер затрат на токены в облаке критически зависит от текущего цикла жизни компании.

*   **Режим «Охотник» (Поиск работы):**
    *   **Активность:** Ежедневный скан аукционных площадок (44-ФЗ, 223-ФЗ), глубокий анализ 5-10 новых лотов в день.
    *   **Облако:** Глубокая экспертиза 10 тендеров в день ≈ **$500 - 1,000 / день**. Поиск контракта в течение месяца может стоить **$15,000 - 30,000** только на «посмотреть».
    *   ** Mac Studio:** Позволяет сканировать рынок «пылесосом», выкачивая и анализируя всё подряд без роста счёта. Это стратегическое преимущество — компания видит больше возможностей, чем конкуренты.
    
*   **Режим «Хранитель» (Сопровождение):**
    *   **Активность:** Когда контракты подписаны на 1-2 года. Работа стабильная: акты (АОСР), письма, проверка соответствия факта проекту.
    *   **Нагрузка:** Снижается до 10-20% от «охотничьей», но остается критически важной для юридической защиты.

### Выводы:
1.  **CAPEX vs OPEX:** Покупка Mac Studio — это разовая инвестиция (CAPEX), которая превращает каждую страницу анализа в бесплатную операцию.
2.  **Агрессивный тендер:** Локальный ИИ позволяет «переваривать» весь рынок тендеров в реальном времени, что финансово невозможно при использовании платных API.
3.  **Окупаемость:** В активной фазе поиска работы (Режим «Охотник») система окупается за **один месяц**.

## 12.5. АКТУАЛИЗАЦИЯ НОРМАТИВНОЙ БАЗЫ (2026)

В 2026 году произошла существенная смена нормативной базы в строительстве. АСД учитывает следующие изменения:

| Старый документ | Новый документ | Статус |
|-----------------|---------------|--------|
| **РД-11-02-2006** | **Приказ Минстроя № 344/пр** | Заменён. Новые правила ведения исполнительной документации |
| **РД 11-05-2007** | **Приказ № 1026/пр** | Заменён. Порядок ведения общего и специальных журналов работ |
| — | **СП 543.1325800.2024** | Новый. Строительный контроль |
| — | **ГОСТ Р 70108-2025** | Новый. Электронная исполнительная документация |

**Важно:** «Росстройконтроль» — это информационная система, **не замена** Государственного строительного надзора (ГСН). ГСН продолжает функционировать в полном объёме. Росстройконтроль обеспечивает электронный документооборот с органом ГСН, но не подменяет его функции.

АСД автоматически обновляет ссылки на нормативную базу при генерации документов (акты, журналы, ИД), чтобы соответствовать актуальным требованиям.

## 12.6. СПЕЦИАЛИЗАЦИЯ КОМПАНИИ

АСД спроектирована под конкретную специализацию строительной компании. Система оптимизирована для следующих видов работ:

### Основные виды работ (поддерживаются всеми агентами)

- **Общестроительные работы** — фундаменты, стены, перекрытия, кровля
- **Бетонные работы** — монолитный бетон, сборный железобетон, опалубка, армирование
- **Земляные работы** — разработка грунта, обратная засыпка, планировка, котлованы
- **Сварочные работы** — сварка металлоконструкций, трубопроводов, закладных деталей
- **Монтажные работы** — монтаж металлоконструкций, ЖБК, оборудования
- **Шпунтовые работы** — погружение/извлечение шпунта (Л5-УМ, Л5-У, Л5-Н), ограждение котлованов

### Исключённые виды работ

Следующие виды работ **не входят** в компетенцию системы и не обрабатываются агентами:

| Исключённый вид работ | Причина исключения |
|----------------------|-------------------|
| **Кабельные работы** | Не профиль — субподряд у специализированных организаций |
| **Слаботочные системы** | Не профиль — проектирование и монтаж сторонними организациями |
| **Вентиляция и кондиционирование** | Не профиль — инженерные системы выполняются специализированными субподрядчиками |
| **Водоснабжение и канализация** | Не профиль — инженерные сети выполняются специализированными субподрядчиками |
| **Инженерные системы** (ОВ, ВК, ЭОМ, ТС) | Не профиль — полный цикл отдают специализированным субподрядчикам |

**Следствие для агентов:** При обнаружении в ПД/ВОР позиций, относящихся к исключённым видам работ, агент ПТО обязан:
1. Пометить такие позиции как «исключённый вид работ»
2. Указать, что данные работы выполняются субподрядчиком
3. Не включать их в сверку ВОР для собственной компании
4. Отразить в отчёте отдельным блоком «Работы субподрядчиков»

## 13. ЧЕГО АСД НЕ БУДЕТ

1. **Не будет чат-ботом общего назначения** — только строительная экспертиза
2. **Не будет заменять Гранд-Смету** — без базы ГЭСН/ФЕР сметы считаются по открытым источникам
3. **Не будет облачным сервисом** — только локально
4. **Не будет автономным агентом** — человек в контуре всегда
5. **Не будет real-time системой** — анализ договора занимает минуты, не секунды
