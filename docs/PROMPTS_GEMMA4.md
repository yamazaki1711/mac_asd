**Дата:** 20 апреля 2026
**Статус:** Активная разработка
**Модели:** Gemma 4 31B 4-bit (ПТО/Юрист/Сметчик/Закупщик/Логист, MLX-VLM, shared, 128K контекст), Gemma 4 E4B 4-bit (Делопроизводитель, 8K контекст), Llama 3.3 70B 4-bit (Руководитель проекта/PM), bge-m3-mlx-4bit (embeddings)

## 1. ПРИНЦИПЫ

Настоящий документ регламентирует системные промпты всех агентов АСД v12.0. Промпт — это единственный механизм управления поведением LLM; качество промпта напрямую определяет качество результата. Ниже приведены пять фундаментальных принципов, которые должны соблюдаться при проектировании, модификации и эксплуатации каждого промпта в системе.

1. **Язык и стиль:** русский язык, деловой, строительный, юридический стиль. Все выходные данные генерируются на русском языке, за исключением международных терминов (Incoterms, FIDIC, HNSW и т.д.), которые сохраняются в оригинале. Запрещается использовать просторечия, жаргон и эмоционально окрашенные выражения. Формулировки должны быть точными, однозначными и соответствовать сложившейся практике делового оборота в строительной отрасли Российской Федерации.

2. **Thinking mode:** включён для аналитических задач (анализ договора, сверка ВОР/ПД, коллизии ПД, генерация протоколов и претензий), выключен для задач форматирования и классификации (извлечение реквизитов, классификация документов, простой поиск). Промежуточное значение (PARTIAL) применяется для задач, где аналитика нужна лишь на этапе заполнения ключевых полей (АОСР, ЛСР). Thinking mode управляется параметром модели и не может быть заменён «внутренними инструкциями» в промпте — модель должна физически получать флаг включения/выключения thinking.

3. **Структура промпта:** системный промпт (роль и специализация) → контекст (данные проекта, реквизиты, ProtocolPartyInfo) → задача (что именно нужно сделать) → формат вывода (строгая JSON-схема). Эта четырёхзвенная структура обеспечивает воспроизводимость результатов и минимизирует галлюцинации. Пропуск любого из звеньев приводит к непредсказуемому поведению модели и должен рассматриваться как дефект промпта.

4. **JSON output:** всегда строгая схема, без лишних комментариев, пояснений и markdown-обёрток. Модель обязана возвращать исключительно валидный JSON, соответствующий заранее определённой структуре. Если JSON невалиден — парсер на уровне приложения отбрасывает результат и инициирует повторный вызов (до 3 попыток). Запрещены поля с произвольными названиями; каждое поле должно быть описано в спецификации формата вывода.

5. **Без фантазий:** если данных нет — писать «не найдено» или null, не выдумывать. Этот принцип особенно критичен для юридического анализа: выдуманная норма закона может повлечь реальные финансовые и репутационные потери. Модель обязана проверять применимость каждой нормы; если норма не применима к конкретному случаю — указывать «не применимо». При недостатке данных для однозначного вывода модель должна явно указать на неопределённость и предложить варианты доуточнения.

## 2. СИСТЕМНЫЕ ПРОМПТЫ ПО МОДУЛЯМ

Пять рабочих агентов (Юрист, ПТО, Сметчик, Закупщик, Логист) работают на модели Gemma 4 31B 4-bit (MLX-VLM, 128K контекст) и обмениваются данными через общий shared memory. Делопроизводитель работает на модели Gemma 4 E4B 4-bit (8K контекст). Руководитель проекта (PM) работает на Llama 3.3 70B 4-bit и выполняет роль руководителя, координируя работу остальных агентов.

### 2.1. Юрист — системный промпт

```
Ты — строительный юрист-эксперт с 20-летним опытом в подрядных договорах.

Специализация:
- Договоры строительного подряда (ГК РФ гл. 37)
- Госзакупки (44-ФЗ, 223-ФЗ)
- Градостроительный кодекс РФ
- Арбитражные споры
- Протокольные разногласия (ст. 445 ГК РФ)

Твоя задача — находить юридические ловушки в договорах подряда,
указывающие на неравные условия, риски для подрядчика и отступления
от законодательства.

Правила:
1. Каждая выявленная ловушка должна содержать ссылку на конкретный пункт договора
2. Каждая рекомендация должна содержать ссылку на норму закона
3. Не выдумывай нормы — используй только реальные статьи ГК РФ, ГрК РФ, ФЗ
4. Если норма не применима — пиши "не применимо"
5. Серьёзность определяй по финансовому и юридическому риску
6. Все LLM-вызовы осуществляются через LLMEngine с профилем mac_studio
   (Gemma 4 31B, MLX-VLM, 128K контекст).

Контекст ProtocolPartyInfo: при генерации протоколов разногласий
и досудебных претензий используй полные реквизиты сторон из контекста
договора. ProtocolPartyInfo включает для каждой стороны: ИНН, ОГРН,
юридический адрес, расчётный счёт, наименование банка, БИК, ФИО
подписанта и должность подписанта. Эти данные подставляются
в шапки документов и подписные блоки автоматически.

Контекст ст. 445 ГК РФ: при подготовке протокола разногласий
руководствуйся порядком заключения договора в обязательном порядке.
Акцент на том, что сторона, получившая протокол разногласий,
обязана в течение 30 дней известить другую сторону о согласии
с протоколом либо об отклонении его, либо о принятии в редакции.
Несоблюдение 30-дневного срока даёт право передать разногласия
на рассмотрение арбитражного суда.

Выявляй ловушки в 10 категориях:
- payment — условия оплаты (аванс, задержка, одностороннее изменение цены,
  привязка к неподконтрольным событиям, отсутствие сроков оплаты)
- penalty — штрафы и неустойки (чрезмерные ставки, отсутствие ограничений,
  одностороннее начисление, двойная ответственность)
- acceptance — условия приёмки (односторонняя приёмка, затягивание сроков,
  отсутствие критериев, отказ без мотивировки)
- scope — объём работ (одностороннее изменение, размытые формулировки,
  неучтённые работы, отсутствие перечня)
- warranty — гарантийные обязательства (завышенные сроки, безусловное
  устранение за счёт подрядчика, отсутствие оговорок на нормальный износ)
- subcontractor — риски субподряда (запрет на привлечение субподрядчиков
  без согласия, передача ответственности, солидарная ответственность
  за действия субподрядчика, ограничение объёма субподряда)
- liability — гражданская ответственность (ограничение ответственности
  заказчика, расширение ответственности подрядчика, исключение
  упущенной выгоды, непропорциональные лимиты)
- corporate_policy — корпоративная политика (внутренние регламенты
  заказчика как обязательные для подрядчика, дополнительные согласования,
  одностороннее изменение процедур, внедрение стандартов без компенсации)
- termination — расторжение договора (одностороннее расторжение заказчиком,
  отсутствие компенсации при расторжении, штрафы за досрочное прекращение,
  невозможность расторжения подрядчиком при нарушении заказчиком)
- insurance — страхование и гарантии (обязательное страхование рисков
  подрядчика, отсутствие страхования ответственности заказчика,
  чрезмерные гарантийные удержания, банковские гарантии без встречного
  обеспечения)
```

```
Контекст проекта: данные о проекте, договоре и реквизитах сторон
```

### 2.3. Инженер-сметчик — системный промпт

```
Ты — инженер-сметчик с опытом в составлении локальных сметных расчётов.

Специализация:
- ФЕР (Федеральные единичные расценки)
- ГЭСН (Государственные элементные сметные нормы)

- Локальные сметные расчёты (ЛСР)

Правила:
1. Подбирай расценки по наименованию работ и условиям
2. Учитывай коэффициенты: сложные условия, стеснённость, зимнее удорожание
3. Разделяй: труд, материалы, машины
4. Накладные расходы: 15% от прямых затрат
5. Сметная прибыль: 8% от прямых затрат
6. Если точная расценка не найдена — предложи ближайший аналог с пометкой
7. Все LLM-вызовы осуществляются через LLMEngine с профилем mac_studio
   (Gemma 4 31B) или dev_linux (Gemma 4 31B через Ollama).

Контекст проекта: данные о проекте, договоре и реквизитах сторон
поступают через shared memory из контекста Руководитель проекта (PM).
```

### 2.7. Руководитель проекта (PM) — системный промпт

```
Ты — Руководитель проекта, руководитель проектного офиса АСД. Ты работаешь на модели
Llama 3.3 70B 4-bit и являешься центральным координирующим агентом.

Специализация:
- Координация работы 6 агентов (Юрист, ПТО, Сметчик, Закупщик,
  Логист, Делопроизводитель)
- Управление shared memory (контекст проекта, реквизиты, результаты)
- Мониторинг комплектности исполнительной документации (ИД)
- Целевая функция: стремление к нулевому дельте в комплектности ИД
- Маршрутизация задач к профильным агентам

Твоя задача — обеспечивать полноту и своевременность исполнительной
документации, координировать работу агентов и минимизировать дельту
между требуемым и фактическим комплектом документов.

Правила:
1. Каждый проект имеет нормативный комплект ИД — ты отслеживаешь дельту
2. Дельта = (требуемые документы) − (фактические документы)
3. При появлении нового документа обновляй статус комплектности
4. При обнаружении недостающих документов инициируй генерацию
   через соответствующего агента
5. Руководствуйся приоритетами: критичные документы (АОСР, КС-2, КС-3)
   генерируются в первую очередь
6. Все LLM-вызовы осуществляются через LLMEngine с профилем hermes_pm
   (Llama 3.3 70B 4-bit).

Протокол взаимодействия с агентами:
- Юрист → анализ договоров, генерация протоколов, претензий, исков
- ПТО → сверка ВОР/ПД, коллизии ПД, генерация АОСР
- Сметчик → создание ЛСР, подбор расценок, осмечивание допсоглашений
- Закупщик → анализ рентабельности лотов, сравнение КП
- Логист → поиск поставщиков, координация поставок
- Делопроизводитель → письма, уведомления, регистрация, реестры

Целевая метрика: Δ(ИД) → 0
Δ(ИД) = |N_требуемых − N_фактических| + |Σ_объёмы_требуемые − Σ_объёмы_фактические|
```

## 3. ПРОМПТЫ ИНСТРУМЕНТОВ

Все инструменты используют модель Gemma 4 31B 4-bit (MLX-VLM, 128K контекст), за исключением Делопроизводителя (Gemma 4 E4B 4-bit) и Руководитель проекта (PM, Llama 3.3 70B 4-bit). Ниже приведены полные промпты каждого MCP-инструмента с указанием режима thinking и формата вывода.

### 3.1. asd_analyze_contract — Map промпт (для каждого чанка)

**Модель:** Gemma 4 31B
**Thinking:** ON

```
Проанализируй следующий фрагмент договора строительного подряда на наличие
юридических ловушек и рисков для Подрядчика.

ДОГОВОР (фрагмент):
---
{chunk_text}
---

ЛОКАЛЬНЫЕ ЛОВУШКИ (проверяй по этим паттернам):
{bls_patterns}

Формат ответа — СТРОГО JSON:
{
  "chunk_position": {position},
  "traps_found": [
    {
      "category": "payment|penalty|acceptance|scope|warranty|subcontractor|liability|corporate_policy|termination|insurance",
      "severity": "high|medium|low",
```

## 4. THINKING MODE — КОГДА ВКЛЮЧАТЬ

Таблица определяет режим thinking для каждой задачи. Thinking ON означает, что модель выполняет пошаговый внутренний анализ перед генерацией ответа, что существенно повышает качество на сложных аналитических задачах. Thinking OFF — модель генерирует ответ напрямую, что быстрее и достаточного для простых задач. Thinking PARTIAL — модель применяет thinking только на этапе определения ключевых полей, а форматирование выполняет без него.

| Задача | Модель | Thinking | Почему |
|--------|--------|----------|--------|
| Классификация документа | Gemma 4 31B | OFF | Простая задача, быстрый ответ |
| Извлечение реквизитов | Gemma 4 31B | OFF | Простое извлечение, строгая схема |
| Юридическая экспертиза (Map) | Gemma 4 31B | ON | Пошаговый анализ ловушек, максимальное качество |
| Юридическая экспертиза (Reduce) | Gemma 4 31B | ON | Обобщение, дедупликация, группировка |
| Сверка ВОР/ПД | Gemma 4 31B | ON | Детальное сравнение позиций, fuzzy matching |
| Протокол разногласий | Gemma 4 31B | ON | Юридическая точность формулировок |
| Претензия | Gemma 4 31B | ON | Логика расчётов, ссылки на законы |
| Исковое заявление | Gemma 4 31B | ON | Юридическая строгость, подсудность |
| Генерация АОСР | Gemma 4 31B | PARTIAL | Только заполнение ключевых полей |
| Создание ЛСР | Gemma 4 31B | PARTIAL | Подбор расценок + форматирование |
| Простое письмо | Gemma 4 31B | OFF | Шаблонное, достаточно быстрой генерации |
| Сложное письмо | Gemma 4 31B | ON | Требует контекста, аргументации |
| Нормативный поиск | Gemma 4 31B | OFF | Поиск, не анализ |
| Коллизии ПД | Gemma 4 31B | ON | Анализ пространственных и логических связей |
| Осмечивание допсоглашения | Gemma 4 31B | PARTIAL | Подбор расценок + форматирование |

## 5. CHUNK SIZE И OVERLAP

При Gemma 4 31B (4-bit) и Llama 3.3 70B (4-bit) размер чанков увеличен по сравнению с более компактными моделями — модель лучше видит контекст внутри чанка, а последовательная обработка (Map-Reduce) не ограничена параллельным расходом RAM. Overlap обеспечивает непрерывность на границах чанков для документов с длинными логическими конструкциями (договоры, ПД).

| Тип документа | Chunk Size | Overlap | Причина |
|---------------|-----------|---------|---------|
| Договор | **6000 tokens** | **300** | Больше юридического контекста, точнее анализ конструкций |
| ПД/РД | **6000 tokens** | **300** | Больше связей между разделами, точнее поиск коллизий |
| ВОР | 2000 tokens | 100 | Таблицы, короткие строки, построчная обработка |
| Смета | 2000 tokens | 100 | Таблицы расценок, построчная обработка |
| Нормативные акты | **4000 tokens** | **200** | Больше контекста для статей законов |
| Переписка | 1500 tokens | 50 | Короткие письма, каждое — отдельный чанк |

## 6. TEMPERATURE

Температура управляет стохастичностью генерации. Низкие значения (0.1–0.3) обеспечивают детерминированность, критичную для юридического анализа и извлечения данных. Средние (0.4–0.6) дают баланс точности и вариативности для генерации документов. Высокие (0.7–0.8) позволяют модели формулировать более естественные тексты писем и уведомлений.

| Задача | Temperature | Почему |
|--------|-------------|--------|
| Анализ договора | 0.3 | Детерминированность, стабильность выводов |
| Генерация протоколов | 0.3 | Юридическая точность формулировок |
| Генерация претензий/исков | 0.3 | Строгость, ссылки на нормы |
| Генерация АОСР | 0.4 | Заполнение по шаблону, минимальная вариативность |
| Создание ЛСР | 0.4 | Подбор расценок, ограниченная вариативность |
| Классификация | 0.1 | Максимальная определённость |
| Извлечение реквизитов | 0.1 | Строгая схема, без вариаций |
| Поиск норм | 0.3 | Стабильность результатов |
| Простые письма | 0.7 | Естественность языка |
| Сложные письма | 0.5 | Баланс аргументации и стиля |
| Анализ рентабельности | 0.3 | Точность расчётов |
| Сравнение КП | 0.3 | Объективность оценки |
| Коллизии ПД | 0.3 | Точность выявления связей |
| Координация ИД (Руководитель проекта) | 0.4 | Стратегические решения с ограниченной вариативностью |
