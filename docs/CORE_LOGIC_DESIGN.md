## 1. Event Manager (State Machine)

LangGraph StateGraph управляет жизненным циклом проекта. **AgentState v2.0** хранит полный контекст конвейера с audit trail, confidence scores и rollback capability. NetworkX Knowledge Graph хранит forensic-связи между документами (сертификаты → партии → АОСР). Каждый переход логируется в PostgreSQL. Это обеспечивает полную трассируемость от тендера до сдачи объекта.

### 1.1. Узлы (Nodes) — События

- `INIT`: Начало жизненного цикла (поиск тендера).
- `TENDER_FOUND`: Закупщик нашел лот.
- `FILES_REGISTERED`: Архив (Делопроизводитель) рассортировал документы.
- `PD_ANALYZED`: ПТО проанализировал ПД.
- `SPEC_EXTRACTED`: ПТО извлек спецификацию.
- `COSTS_ESTIMATED`: Сметчик сформировал ЛСР.
- `CONTRACT_ANALYZED`: Юрист проверил договор по БЛС (61 ловушка).
- `PROTOCOL_GENERATED`: Юрист сгенерировал протокол разногласий с ProtocolPartyInfo.
- `VENDORS_SOURCED`: Логист нашел поставщиков.
- `QUOTES_COMPARED`: Логист выбрал КП.
- `SUPPLEMENT_ESTIMATED`: Сметчик осметил допсоглашение.
- `ID_COMPLETENESS_CHECKED`: ПТО проверил комплектность ИД (delta рассчитан).
- `ARCHIVED`: Архив сохранил финальный пакет.
- `PROJECT_WON`: Тендер выигран, старт СМР.
- `DEADLINE_MISSED`: Просрочка (триггер для Юриста → претензия).
- `CLAIM_GENERATED`: Претензия сгенерирована.
- `LAWSUIT_GENERATED`: Иск сгенерирован.

## 2. Агент Логист: Детальная логика

Логист — самый сложный агент в плане взаимодействия с внешней средой. Он координирует поиск поставщиков, формирование запросов, получение и сравнение коммерческих предложений. Все LLM-операции выполняются через Gemma 4 31B.

### 2.1. Цикл работы Логиста (Inner Loop)

1. **Анализ спецификации:** Получает от ПТО список ТМЦ (напр. «Шпунт Л5-УМ, 150т»). Gemma 4 31B извлекает структурированные данные (наименование, количество, единица измерения, ГОСТ/ТУ).
2. **Поиск (Web Search):** Ищет актуальных поставщиков в регионе через `WebSearch` MCP. Gemma 4 31B оценивает релевантность результатов и фильтрует неквалифицированных поставщиков.
3. **Формирование RFQ:** Генерирует PDF/Email запрос с спецификацией и требованиями. Использует ProtocolPartyInfo для реквизитов отправителя.
4. **Коммуникация:** Имитация или реальная отправка через инструменты интеграции (Email/Telegram).
5. **Сбор данных:** Ожидание входящих писем, парсинг КП через `asd_parse_price_list`. Для сканированных КП используется OCR-конвейер (pytesseract → Gemma 4 31B VLM).
6. **Сравнение:** Формирование сравнительной таблицы (Конкурентный лист) через `asd_compare_quotes`. Gemma 4 31B оценивает не только цену, но и сроки доставки, условия оплаты и надёжность поставщика из базы vendors.

## 4. OCR Стратегия (Mac Studio Fallback)

OCR-конвейер АСД v12.0: PyMuPDF (текстовый слой) → rapidocr (скан-копии) → tesseract (fallback). Для сложных документов зарезервирован PaddleOCR PP-OCRv5 (109 языков, +13% точности).

### 4.1. Tier 1 (Base): pytesseract

Применяется для простых документов: машинописный текст, стандартные формы без графических элементов. Быстрый и экономичный по памяти. Используется по умолчанию для всех PDF с текстовым слоем.

- **Скорость:** ~1-3 сек/страница
- **RAM:** Минимально (<100 MB)
- **Качество:** Хорошее для машинописного текста, низкое для таблиц и рукописных пометок

### 4.2. Tier 2 (Vision): Gemma 4 31B VLM

Применяется для сложных документов: таблицы с множеством колонок, печати, рукописные пометки, сканы низкого качества, чертежи с текстовыми выносками. Gemma 4 31B VLM уже загружена как shared model (23 GB) и доступна всем агентам без дополнительных затрат RAM.

- **Скорость:** ~10-30 сек/страница
- **RAM:** ~5 GB (on-demand, выгружается после использования)
- **Качество:** Высокое для таблиц, печатей, рукописного текста
- **Критерий активации:** Tier 1 confidence < 0.7 или обнаружение таблиц/печатей


## 5. Evidence Graph v2 — Единый граф для двух режимов (✅ Реализован, май 2026)

### 5.1. Архитектура

Evidence Graph v2 — центральная структура данных ASD. Заменяет собой изолированные проверки forensic KAG на единую модель с confidence и inference.

**Фундаментальный принцип:** ПД/РД есть всегда. ASD не начинает «с пожара» — она начинает с того, что должно быть построено.

**ProjectLoader** (нулевой слой) парсит ПД/РД и создаёт baseline:
- WorkUnit'ы (PLANNED) из спецификаций и ВОР
- Volume на каждый WorkUnit
- Location-иерархию из генплана
- MaterialBatch (ожидаемые поставки) из сметы
- TEMPORAL_BEFORE из ПОС/ППР
- REFERENCED-документы (АОСР, сертификаты, ИС) — что должно быть

**Два режима от одного baseline:**

| | Сопровождение | Антикризис |
|---|---|---|
| WorkUnit'ы | PLANNED → IN_PROGRESS → COMPLETED (агенты) | PLANNED → INFERRED (inference) |
| confidence | 1.0 | 0.4–0.85 → HITL → 1.0 |
| Journal | Заполняется по факту | Реконструируется |

### 5.2. Inference Engine

6 symbolic-правил (не LLM). Пример:
- ТТН: поставка 15.03, 100 шт шпунта
- typical_rate: 12 шт/день
- → INFER: погружение 16.03–23.03 (confidence 0.85)

Библиотека темпов в `data/typical_rates.yaml` (10 видов работ).

### 5.3. Модули

- `src/core/evidence_graph.py` — 7 типов узлов, 11 типов связей, confidence, GML-персистенция
- `src/core/inference_engine.py` — 6 правил вывода, typical_rates
- `src/core/project_loader.py` — парсинг ПД/РД → baseline
- Архитектура: `.hermes/plans/2026-05-02_evidence-graph-v2-architecture.md`

Триггерная цепочка WorkEntry → AOSR автоматизирует создание актов освидетельствования скрытых работ на основе записей журнала работ.

### 7.1. Архитектура цепочки


## 5. Руководитель проекта как руководитель (PM Role)

Руководитель проекта на базе **Llama 3.3 70B 4-bit** — это не просто маршрутизатор вызовов инструментов. Это **активный руководитель**, который непрерывно стремится к нулевому delta в комплектности ИД и максимальному покрытию рисков. Роль Руководитель проекта реализована через набор стратегических принципов и механизмов контроля.

### 5.1. Принцип Zero Delta

Руководитель проекта отслеживает **delta** — разрыв между требуемой и имеющейся исполнительной документацией. Delta рассчитывается через `asd_id_completeness` и обновляется при каждом значимом событии (генерация акта, получение документа, завершение этапа работ).

- **Delta = 0:** Проект полностью укомплектован ИД. Можно передавать Заказчику.
- **Delta > 0:** Есть недостающие документы. Руководитель проекта инициирует цикл дозаполнения: назначает ПТО генерацию недостающих актов, Делопроизводителя — сбор недостающих паспортов/сертификатов.
- **Delta увеличивается:** Красный флаг. Новые работы выполнены без оформления. Руководитель проекта приостанавливает задачу и требует оформления.

### 5.2. Принцип полного покрытия БЛС

При анализе договора Руководитель проекта проверяет, что **все 61 ловушка БЛС** проверены Юристом, и что все ловушки с severity ≥ high отражены в протоколе разногласий. Если Юрист не покрыл какую-либо критическую ловушку — Руководитель проекта инициирует повторный анализ с явным указанием непокрытой ловушки.

### 5.3. Принцип кризисного управления

При наступлении кризисного события (просрочка оплаты, уклонение от приёмки, отказ подписывать акты) Руководитель проекта автоматически:

1. Приоритизирует претензионный конвейер над всеми другими задачами.
2. Инициирует `asd_generate_claim` с автоматическим расчётом неустойки.
3. При неудовлетворении претензии > 30 дней — запускает `asd_generate_lawsuit`.
4. Фиксирует все дедлайны через `asd_track_deadlines` и уведомляет пользователя.

### 5.4. Принцип управления памятью

При достижении критических порогов RAM Руководитель проекта принимает решения об оптимизации:

| Давление памяти | Действие Руководитель проекта |
|-----------------|-----------------|
| Low (<70%) | Все модели загружены, Map-Reduce активен |
| Critical (>90%) | Только Руководитель проекта + Gemma 4 31B, Gemma 4 E4B выгружена |

## 6. Map-Reduce Pipeline для LegalService

Для обработки длинных договоров (≥6K символов) LegalService использует **Map-Reduce pipeline** — двухфазный процесс, который позволяет анализировать документы произвольной длины без потери качества. Pipeline реализован поверх Gemma 4 31B и работает через LLMEngine.

### 6.1. Фаза Map (Разбиение и параллельный анализ)

Договор разбивается на перекрывающиеся чанки размером ~6K символов с перекрытием 500 символов для сохранения контекста на границах. Каждый чанк обрабатывается **независимо** вызовом Gemma 4 31B с промптом, ориентированным на выявление:

- Ловушек БЛС (61 ловушка в 10 категориях)
- Противоречий между разделами
- Отсутствующих обязательных условий (ФЗ-44 ст. 34)
- Неоднозначных формулировок

Параллелизм Map ограничен доступной памятью: при low pressure — до 4 параллельных вызовов, при medium — 2, при high — 1 (последовательно).

```
Договор (150 стр, 185K символов)
  → Чанк 1 (0-6K): {...findings: [...]}
  → Чанк 2 (5.5K-11.5K): {...findings: [...]}
  → Чанк 3 (11K-17K): {...findings: [...]}
  → ...
  → Чанк 12 (66K-72K): {...findings: [...]}
```

### 6.2. Фаза Reduce (Синтез и дедупликация)

Результаты всех чанков собираются и обрабатываются единым вызовом Gemma 4 31B с промптом Reduce, который:

1. **Дедуплицирует** findings — один и тот же риск может быть выявлен в нескольких чанках из-за перекрытия.
2. **Выявляет противоречия** между разделами — только на этом этапе видны кросс-разделные несоответствия (например, срок в разделе 3 противоречит сроку в разделе 7).
3. **Рассчитывает итоговый verdict** — approved / approved_with_comments / rejected.
4. **Формирует итоговый summary** с количеством замечаний по категориям.

### 6.3. Quick Review vs Map-Reduce

| Параметр | Quick Review | Map-Reduce |
|----------|-------------|------------|
| Порог | <6K символов | ≥6K символов |
| LLM-вызовы | 1 | N чанков + 1 Reduce |
| Время | ~30 сек | ~4-10 мин |
| Память | Минимум | Пропорционально чанкам |
| Качество | Хорошее для коротких | Максимальное для длинных |
| Противоречия | Только в рамках чанка | Кросс-разделные |

### 6.4. Интеграция с БЛС

Каждый finding из Map-Reduce автоматически сопоставляется с БЛС через fuzzy matching по legal_basis. Если legal_basis в finding совпадает со статьёй в БЛС (например, «ст. 743 ГК РФ» → ловушка №4 «Одностороннее изменение объёма»), в результат добавляется блок `blc_match` с ID, названием и критичностью ловушки. Это обеспечивает полную трассируемость от выявленного риска до конкретной ловушки в библиотеке.

## 7. ProtocolPartyInfo: Извлечение реквизитов сторон

ProtocolPartyInfo — это структурированный блок данных, содержащий полные реквизиты Заказчика и Подрядчика, извлечённые из текста договора. Блок используется во всех генерируемых документах (протокол разногласий, претензия, исковое заявление, акты, письма) для обеспечения юридической корректности и соответствия реквизитов.

### 7.1. Структура ProtocolPartyInfo

```python
class ProtocolPartyInfo(BaseModel):
    """Реквизиты стороны договора для генерации документов."""
    name: str              # Наименование организации
    inn: str               # ИНН (10 или 12 цифр)
    ogrn: str              # ОГРН (13 или 15 цифр)
    legal_address: str     # Юридический адрес
    checking_account: str  # Расчётный счёт (20 цифр)
    bank: str              # Наименование банка
    bik: str               # БИК (9 цифр)
    correspondent_account: str | None = None  # Корр. счёт
    signatory: str         # ФИО подписанта
    position: str          # Должность подписанта
    basis: str | None = None  # Основание (Устав, доверенность)
```

### 7.2. Процесс извлечения

Извлечение ProtocolPartyInfo выполняется в три этапа:

1. **Первичный поиск:** Gemma 4 31B получает полный текст договора и извлекает блоки с реквизитами через специализированный промпт. Модель обучена распознавать стандартные формулировки: «Заказчик:», «Подрядчик:», «Исполнитель:», «Реквизиты сторон:» и т.д.

2. **Валидация:** Извлечённые данные проходят программную валидацию:
   - ИНН: контрольная сумма (10 или 12 цифр)
   - ОГРН: контрольная цифра (13 или 15 цифр)
   - БИК: 9 цифр, начинается с 04... или 02...
   - Расчётный счёт: 20 цифр
   - Наименование: непустое

3. **Fallback при неполных данных:** Если какие-либо поля не найдены в тексте договора, Gemma 4 31B выполняет повторный поиск с расширенным контекстом (включая приложения к договору). Если поле всё ещё не найдено — оно помечается как `null` и пользователь уведомляется о необходимости ручного заполнения.

### 7.3. Использование в документах

ProtocolPartyInfo автоматически подставляется в следующие генерируемые документы:

| Документ | Поля ProtocolPartyInfo |
|----------|----------------------|
| Протокол разногласий | Все поля обеих сторон |
| Претензия | name, inn, legal_address, signatory, position |
| Исковое заявление | Все поля обеих сторон + basis |
| АОСР | name, signatory, position |
| Сопроводительное письмо | name, legal_address, inn |
| Реестр отправки | name, inn |

### 7.4. Кэширование

Извлечённый ProtocolPartyInfo кэшируется в PostgreSQL (таблица `contract_parties`) и привязывается к `contract_id`. При повторной генерации документов для того же контракта реквизиты не извлекаются заново, а берутся из кэша. При обновлении текста договора (новая редакция) кэш инвалидируется автоматически.

## 9. Скрытые работы → Шлейф документов (из Пособия по ИД Вып.2)

**Источник:** Щербаков, Пособие по ИД Выпуск №2 (425 стр.), Приказ 344/пр, СП 543.1325800.2024.

Данный раздел задаёт **единственный источник истины** для Chain Builder и HITL System: какие документы ASD обязана требовать/восстанавливать для каждого вида скрытых работ.

### 9.1. Принцип

Скрытая работа — работа, влияющая на безопасность ОКС, контроль которой невозможен после выполнения последующих работ. Каждой скрытой работе соответствует **обязательный документальный шлейф**, без которого АОСР недействителен:

```
АОСР = ВОР + ИС + документы качества + протоколы испытаний + уведомление за 3 р.д.
```

### 9.4. Уровни уверенности для Chain Builder

При реконструкции ИД Chain Builder назначает confidence документу шлейфа:

| Источник документа | Confidence | Статус |
|-------------------|-----------|--------|
| Оригинал с подписями (в папке) | 1.0 | ORIGINAL |
| Скан с подписями (VLM подтвердил) | 0.9 | SCAN |
| Восстановлен из косвенных данных (Inference Engine) | 0.5–0.8 | RECONSTRUCTED |
| Упомянут в другом документе, но отсутствует | 0.3 | REFERENCED |
| Требуется по нормативу, не найден нигде | 0.0 | MISSING |

### 9.5. HITL-приоритеты

HITL-система задаёт вопросы оператору в порядке важности:

1. **CRITICAL** — отсутствует АОСР при наличии материала/КС-2 (работа выполнена, но не освидетельствована)
2. **HIGH** — отсутствует сертификат на материал, использованный по КС-2
3. **MEDIUM** — отсутствует исполнительная схема при наличии АОСР
4. **LOW** — отсутствует спецжурнал при наличии АОСР (формальность)
