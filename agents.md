# AGENTS.MD — Манифест Оркестратора АСД v12.0 (Полный Жизненный Цикл)

> Этот файл — главная инструкция для PM-агента (оркестратора) по управлению субагентами.
> PM читает этот файл при каждом запуске для определения логики маршрутизации.
>
> ⚠️ **Исторически**: в ранних версиях ASD было два оркестратора — `hermes` (тендеры) и `pm` (документы).
> В мае 2026 они объединены: лучшая тендерная логика перенесена в PM, hermes_router.py удалён.
> Режим «поиск тендеров для КСК» — технический долг, отложен до Mac Studio и продакшен-готовности.

## 2. Протокол "Полный Жизненный Цикл ОКС" (Event-Driven Workflow)

Система управляется событиями (Events). Каждый агент подписан на определенные триггеры.

### 2.1. ЭТАП 1: Предконтрактная подготовка (Тендер)

**Триггер:** Плановый запуск Закупщика
1. **Закупщик** находит лот, проверяет лимита НМЦК (по прошлым КС-11 из БД и MaterialCatalog), скачивает архивы (ПД, ВОР, ЛСР).
2. **Делопроизводитель** распаковывает архивы и структурирует файлы в `ЗАКУПКИ/{project_id}/`.
3. **ПТО** анализирует чертежи, извлекает спецификации материалов. Возвращает проверенную ВОР.
4. **Логист** берет спецификацию от ПТО, находит поставщиков (арматура, шпунт и т.д.), выбирает 3 лучших КП.
5. **Сметчик** и **Юрист** ПАРАЛЛЕЛЬНО проверяют ВОР/сметы (на рентабельность) и Контракт (на БЛС ловушки — 58 ловушек в 10 категориях).
6. **PM** формирует итоговый вердикт (гибридная 3-стадийная модель: weighted scoring + LLM reasoning + veto rules).

### 2.2. ЭТАП 2: Снабжение и Входной контроль

**Триггер:** Событие `ProjectWon` (Выигрыш тендера)
1. **Логист** оформляет финальные заявки выбранным поставщикам, контролирует поступление ТТН и сертификатов качества.
2. **Делопроизводитель** вносит сертификаты и паспорта качества в базу (инструмент `asd_register_document`).
3. **ПТО** автоматически генерирует "Акт входного контроля".

### 2.3. ЭТАП 3: Производство и Исполнительная документация (ИД)

**Триггер:** Событие `OJPEntryClosed` (В ОЖР закрыта запись о выполнении работ)
1. **ПТО** получает сигнал. Ищет связанные сертификаты материалов (от Логиста) и чертежи РД.
2. **ПТО** вызывает `asd_generate_act` для создания АОСР (Акта скрытых работ).
3. **Делопроизводитель** присваивает АОСР номер и сохраняет в реестр ИД.

### 2.4. ЭТАП 4: Коммерческое закрытие (Ежемесячно)

**Триггер:** Конец отчетного расчетного периода
1. **Сметчик** собирает все подписанные АОСР за месяц. Сверяет со сметой (`asd_estimate_compare`).
2. **Сметчик** ведет журнал КС-6а, формирует заготовки актов КС-2 и КС-3.
3. **Делопроизводитель** формирует опись для отправки (`asd_prepare_shipment`).

### 2.5. ЭТАП 5: Претензии и Завершение

**Триггер:** Событие `DeadlineExpired` (Нарушение сроков оплаты или ответа)
1. **Юрист** генерирует досудебную претензию (`asd_generate_claim`).
2. **Юрист** формирует исковое заявление (`asd_generate_lawsuit`) при необходимости.
3. **ПТО** собирает итоговый реестр ИД (`asd_id_completeness`) для подписания КС-11.

## 2.6. ЛАБОРАТОРНЫЙ КОНТРОЛЬ (lab_pipeline)

Манифест v3.0 добавляет `lab_pipeline` — конвейер работы с строительными лабораториями. Участвуют 4 агента:

```
lab_pipeline:
  1. ПТО: lab_control_plan_create → lab_sample_register → lab_report_review
  2. Закупщик: lab_organization_search → lab_quote_request → lab_quote_compare
  3. Логист: lab_sample_delivery
  4. Делопроизводитель: lab_request_letter_generate → lab_contract_register → lab_act_register → lab_report_file
  Helper: lab_register_report, lab_request_update_status
```

**Полный workflow:**
План контроля → Поиск лабораторий → Запрос КП → Сравнение → Договор → Отбор образцов → Доставка → Акт → Отчёт → Подшивка в ИД

**Виды испытаний:**
- Сварочные НДК: ВИК, МПК, УЗК, РК, Капиллярный
- Бетонные: сжатие, изгиб, морозостойкость, водонепроницаемость

**Триггер:** Событие `LabControlRequired` (необходимость лабораторных испытаний по плану контроля)

## 3. Обработка ошибок в конвейере

| Ситуация                           | Действие PM / Система                                      |
|------------------------------------|--------------------------------------------------------------|
| Нет сертификата на материал в БД   | Делопроизводитель блокирует создание АОСР, уведомляет РП.    |
| Смета ≠ ВОР                        | Конфликт фиксируется, Сметчик возвращает задачу в ПТО.       |
| Юрист нашёл критическую ловушку    | Остановка подписания контракта (Уровень риска: HIGH).        |
| Логист не нашел КП с нужной ценой  | Эскалация на PM для пересмотра бюджета или отказа.         |
| Субагент вернул пустой результат   | Retry (max 2), затем эскалация на человека.                  |
| ПТО не нашел шпунт на чертеже      | Пометка "шпунт не обнаружен — требует ручной проверки".      |

## 4. Хранение состояния

* **NetworkX** — граф проектов, документов, событий (in-memory + файловая сериализация в data/graphs/).
* **PostgreSQL (pgvector)** — реляционные данные, векторные эмбеддинги, БЛС (58 ловушек в 10 категориях), прайс-листы.
* **Redis** — УДАЛЁН. Заменён in-process кэшем (cachetools). Экономия 2GB RAM.
* **LLMEngine** — единый интерфейс к MLX-бэкенду (Mac Studio M4 Max).
* **Artifact Store** — файлы сохраняются в `./data/artifacts/`.
* **Google Workspace** — Sheets (табличные данные, ВОР, журналы), Docs (шаблоны актов), Drive (файловое хранилище).
* **Registry** — `data/artifacts/{project_id}/registry.json`

## 5. Технические регламенты

* **MLX** — единый движок для всех LLM-операций (chat, vision, embeddings). Профиль `mac_studio` по умолчанию.
* **Shared models:** Gemma 4 31B — одна физическая копия в памяти для ПТО, Сметчика, Юриста, Закупщика и Логиста (128K контекст).
* **Qwen3.5-27B** — УДАЛЕНА из модельного ряда v12.0. Все рабочие агенты переведены на Gemma 4 31B.
* **Google Workspace** — интеграция через OAuth2 токен Hermes Agent (~/.hermes/google_token.json) или Service Account. Папки Drive, шаблоны Docs/Sheets настраиваются через `.env`.

## 6. Правила Orchestrator'а (PM)

1. **Конвейер событий (Event Bus):** PM работает как слушатель событий и диспетчер, а не как жесткий скрипт.
2. **Гибридная модель решений (v12.0):** Взвешенное голосование агентов (Юрист 0.35, Сметчик 0.25, ПТО 0.20, Закупщик 0.12, Логист 0.08) → LLM reasoning для серой зоны → Veto rules.
3. **Логирование:** Каждое действие агента (созданный АОСР, полученное КП) = запись в граф БД с timestamp.
4. **Идемпотентность:** Повторная генерация документа (например, при правках) не дублирует узлы, а версионирует их.
5. **Человек в контуре:** Прохождение ключевых "ворот" (Gateways) ПТО -> КС-2 -> Оплата требует верификации человеком-руководителем.
6. **Профилирование:** Все LLM-вызовы проходят через LLMEngine, который направляет запросы в MLX-бэкенд.
