## 2. Протоколы взаимодействия

### 2.1. Полный поток: Документ → ВОР → Смета → Виза → Лоты → Архив

```
┌──────────┐     ВОР       ┌──────────┐    Смета     ┌──────────┐
│  ПТО     │ ───────────► │ Сметчик  │ ───────────► │  Юрист   │
│ (Gemma4) │              │ (Qwen3)  │              │(DeepSeek)│
└────┬─────┘              └────┬─────┘              └────┬─────┘
     │                         │                         │
     │         ВОР             │         Смета           │    Заключение
     │   (volumes JSON)        │     (rates + totals)    │   (verdict JSON)
     │                         │                         │
     ▼                         ▼                         ▼
┌─────────────────────────────────────────────────────────────────┐
│                        Neo4j Graph DB                           │
│  (:Project)-[:HAS_DRAWING]->(:Drawing)                          │
│           -[:HAS_VOR]->(:VOR)                                   │
│           -[:HAS_ESTIMATE]->(:Estimate)                         │
│           -[:HAS_LEGAL_REVIEW]->(:LegalOpinion)                 │
│           -[:HAS_LOTS]->(:ProcurementLot)                       │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
                        ┌──────────────┐
                        │   Закупщик   │ ── лоты ──► Делопроизводитель
                        │  (Qwen3)     │              (архивация)
                        └──────────────┘
```

### 2.2. Роль Делопроизводителя в потоке

Делопроизводитель участвует на **каждом этапе**:

1. **Входящий документ** → Делопроизводитель регистрирует, присваивает номер и версию
2. **Передача между агентами** → Делопроизводитель обновляет статус документа
3. **Результат** → Делопроизводитель архивирует финальную версию
4. **Реестр** → Все узлы в Neo4j синхронизированы с `registry.json`

### 2.3. Протокол: ПТО → Сметчик (передача ВОР)

**Условие запуска:** ПТО завершил Vision Cascade, все tiles обработаны, ВОР агрегирована.

**Действия Hermes:**

1. Принять ВОР от ПТО, проверить:
   * Все tiles обработаны (`metadata.tiles_processed == expected`)
   * Нет позиций с confidence < 0.3 без пометки "требует проверки"
2. Передать Делопроизводителю: зарегистрировать ВОР, статус → "in_progress"
3. Создать узел `(:VOR)` в Neo4j, сохранить JSON ВОР
4. Создать задачу для Сметчика:

   ```yaml
   task:
     to: smeta
     type: estimate
     payload:
       vor_ref: "<neo4j_node_id>"
       project_id: "<project_uuid>"
       requirements:
         - "Использовать актуальный индекс Минстроя"
         - "Помечать расценки 'по аналогии'"
   ```

5. Дождаться результата, проверить структуру сметы

### 2.4. Протокол: Сметчик → Юрист (визирование тендера)

**Условие запуска:** Смета собрана, totals рассчитаны.

**Действия Hermes:**

1. Принять смету, проверить:
   * Все позиции ВОР покрыты расценками
   * `grand_total` рассчитан корректно
2. Передать Делопроизводителю: зарегистрировать смету
3. Создать узел `(:Estimate)` в Neo4j
4. Создать задачу для Юриста:

   ```yaml
   task:
     to: legal
     type: legal_review
     payload:
       estimate_ref: "<neo4j_node_id>"
       project_id: "<project_uuid>"
       review_scope:
         - "Проверить НМЦК на соответствие ФЗ-44 ст. 22"
         - "Проверить требования к поставщикам"
         - "Проверить условия контракта (ст. 34 ФЗ-44)"
   ```

5. При verdict = "approved" или "approved_with_comments" → передать Закупщику
6. При verdict = "rejected" → вернуть Hermes с рекомендациями, цикл повторить

### 2.5. Протокол: Юрист → Закупщик → Делопроизводитель (архивация)

**Условие запуска:** Юрист дал визу (verdict ≠ rejected), Закупщик сформировал лоты.

**Действия Hermes:**

1. Записать `(:LegalOpinion)` в Neo4j
2. Создать задачу для Закупщика:

   ```yaml
   task:
     to: procurement
     type: form_lots
     payload:
       estimate_ref: "<neo4j_node_id>"
       legal_ref: "<legal_opinion_id>"
       project_id: "<project_uuid>"
   ```

3. Принять лоты, проверить НМЦК, сохранить `(:ProcurementLot)`
4. Передать Делопроизводителю: архивировать весь пакет документов
5. Финальный статус проекта → "ready_for_tender"

## 3. Обработка ошибок

| Ситуация                           | Действие Hermes                                        |
|------------------------------------|--------------------------------------------------------|
| Субагент вернул пустой результат   | Retry (max 2), затем эскалация на человека             |
| Субагент вернул ошибку формата     | Reject с комментарием, retry 1 раз                     |
| Конфликт данных (смета ≠ ВОР)      | Запросить ПТО пересмотреть, затем Сметчику повторно    |
| Юрист отклонил дважды              | Эскалация на человека + лог в граф                     |
| Таймаут MCP (> 120s)               | Retry с увеличенным timeout, лог                       |
| Конфликт версий документов         | Делопроизводитель блокирует, уведомляет Hermes         |

## 4. Хранение состояния

* **Neo4j** — узлы проектов, документы, задачи, результаты
* **Redis** — кеш vision-results, rate-lookups, RAG-queries (TTL 24h)
* **Artifact Store** — файлы (чертежи, ВОР, сметы, заключения) в `./data/artifacts/`
* **LightRAG** — векторные индексы для RAG (legal_rag, rates_rag)
* **Registry** — `data/artifacts/{project_id}/registry.json` (делопроизводитель)

---

## 5. Правила Orchestrator'а

1. **Один поток** — Hermes ведёт один проект за раз (параллелизм только внутри pipeline)
2. **Логирование** — каждое действие агента = запись в граф с timestamp
3. **Идемпотентность** — повторный запуск с теми же данными не должен дублировать узлы
4. **Человек в контуре** — любая эскалация = уведомление + пауза до подтверждения
5. **Версионность** — все документы проходят через Делопроизводителя, никаких прямых записей в файлы
