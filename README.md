# MAC ASD v12.0 — Мультиагентная ИИ-система строительного субподряда

**MAC ASD** (Multi-Agent Construction AI System for Automated Subcontracting Decisions) — полностью локальная, оффлайн-ориентированная система автоматизации создания и восстановления **полного комплекта исполнительной документации (ИД)** на объекты капитального строительства (ОКС) любой сложности и любых видов работ. Спроектирована для запуска на **Mac Studio M4 Max 128GB** с использованием нативного фреймворка **MLX** для инференса LLM.

> Не заменяет специалистов. **Усиливает** их. Человек всегда принимает финальное решение.

## Архитектура: 7 специализированных агентов

Система построена на базе **LangGraph StateGraph** — каждый агент является узлом графа, а оркестратор **Руководитель проекта** маршрутизирует задачи между ними через **HermesRouter** (гибридная 3-стадийная модель решений). Все LLM-вызовы проходят через единый **LLMEngine** с поддержкой профилей (`dev_linux` / `mac_studio`).

## Два режима работы

### 1. Тендерный конвейер (lot_search)
Мониторинг тендерных площадок → регистрация документов → оценка НМЦК → извлечение ВОР → логистика → сметный расчет → юридическая экспертиза → вердикт Hermes → рефлексия.

### 2. Сопровождение строительства (construction_support)
Вводный контроль → оперативный контроль объёмов → финансовый контроль (КС-2/КС-3) → логистическое обеспечение → претензионная работа → завершение объекта.

### Лабораторный конвейер (sub-pipeline)
ПТО: план контроля → Закупщик: поиск лабораторий / запрос КП → Делопроизводитель: письмо-обращение → Закупщик: сравнение КП → Юрист: проверка договора → Делопроизводитель: регистрация договора → ПТО: отбор образцов → Логист: доставка → Делопроизводитель: акт → ПТО: проверка заключения → Дело: подшивка.

## Ключевые компоненты

### WorkTypeRegistry — единый контракт (SSOT)
Модуль `src/agents/skills/common/work_type_registry.py` обеспечивает:
- `WORK_TYPE_TO_SMETA_CATEGORY` — маппинг 20 видов работ → 10 сметных категорий
- `WORK_TYPE_TO_LEGAL_WORK_TYPE` — маппинг 20 видов работ → 10 юридических категорий
- `WORK_TYPE_TO_FER_PREFIX` — маппинг 20 видов работ → префиксы ФЕР
- `get_smeta_category()`, `get_legal_work_type()`, `get_fer_prefix()` — функции разрешения
- `resolve_work_type()` — fuzzy-разрешение произвольной строки в WorkType
- `list_all_work_types()` — полный перечень с кросс-ссылками

### HermesRouter — гибридная 3-стадийная модель решений
1. **Weighted Scoring:** Агентные веса (Юрист 0.35, Сметчик 0.25, ПТО 0.20, Закупщик 0.12, Логист 0.08). Score ≥ 0.7 → GO, ≤ 0.3 → NO GO.
2. **LLM Reasoning:** Для серой зоны (0.3–0.7) — Llama 3.3 70B принимает решение с обоснованием.
3. **Veto Rules:** Автоматический NO GO при: legal=DANGEROUS, margin<10%, critical_traps≥3, НМЦК<70% рынка.

### AgentState v2.0 — типизированное состояние конвейера
- `VORResult`, `LegalResult`, `SmetaResult`, `ProcurementResult`, `LogisticsResult`, `ArchiveResult` — типизированные sub-структуры
- `audit_trail` — пошаговый лог выполнения (StepLog)
- `confidence_scores` — уверенность агентов в результатах
- `hermes_decision` — структурированное решение Hermes (HermesDecision)
- `revision_history` — история ревизий для rollback
- `create_initial_state()` — фабрика начального состояния
- `migrate_v1_to_v2()` — миграция из старой схемы

### БЛС — Библиотека Ловушек Субподрядчика
**21 структурированная ловушка** из судебной практики в seed-данных (`contract_risks.py`), покрывающая все 10 категорий работ. Дополнительно — 58 ловушек в 10 категориях в YAML-каталоге (`traps/default_traps.yaml`). Хранится в PostgreSQL с векторными эмбеддингами (pgvector), что позволяет Юристу осуществлять семантический поиск аналогичных ситуаций через RAG.

### Map-Reduce для анализа контрактов
Длинные контракты (100+ страниц, свыше 128K токенов) разбиваются на чанки (6000 tokens, overlap 300), каждый анализируется Юристом отдельно (map), затем результаты агрегируются (reduce). Для большинства контрактов (до 300K символов / ~128K токенов) используется Quick Review без Map-Reduce — полный контракт помещается в контекст Gemma 4 31B.

### Цикл рефлексии (самообучение)
После каждого завершенного пайплайна система анализирует аудит-логи и обновляет wiki-правила агентов. Система умнеет с каждым обработанным лотом.

### Опытный контур — Lessons Learned (3 уровня)
Институциональная память системы, построенная на анализе реальных лотов:

1. **Level 1: База уроков** — PostgreSQL + pgvector. Каждый урок содержит: лот, вид работ, агент, категория, критичность, нормативка, контекст лота, эмбеддинг. Seed-данные из лота №0138300000426000314 (снос МКД, Петропавловск-Камчатский).
2. **Level 2: RAG-инъекция** — Автоматический поиск релевантных уроков через pgvector при анализе нового лота. Уроки подставляются прямо в промпт агента через `asd_lessons_context`.
3. **Level 3: Skill Mutation** — После N подтверждений (порог=3) урок мутирует в автоматическое правило, которое постоянно включено в промпт агента без RAG-поиска.

Категории уроков: `coeff_error`, `risk`, `false_risk`, `norm_violation`, `contract_trap`, `best_practice`.

Сервис: `src/core/lessons_service.py` (449 строк). MCP-инструменты: 7 штук (search, add, verify, list, context, reject, auto_rules).

## Быстрый старт

### 1. Инфраструктура
```bash
cd infrastructure
docker compose up -d    # PostgreSQL
```

### 2. Настройка окружения
```bash
cp .env.example .env   # Указать ASD_PROFILE, пути к БД
```

### 3. Запуск системы
```bash
source .venv/bin/activate
export PYTHONPATH=$PYTHONPATH:.
export ASD_PROFILE=dev_linux    # или mac_studio
python src/main.py
```

### 4. Запуск MCP-сервера
```bash
python -m mcp_servers.asd_core.server
```
