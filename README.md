# MAC ASD v11.3 — Мультиагентная ИИ-система строительного субподряда

**MAC ASD** (Multi-Agent Construction AI System for Automated Subcontracting Decisions) — полностью локальная, оффлайн-ориентированная система автоматизации работы департамента строительства. Спроектирована для запуска на **Mac Studio M4 Max 128GB** с использованием нативного фреймворка **MLX** для инференса LLM.

> Не заменяет специалистов. **Усиливает** их. Человек всегда принимает финальное решение.

---

## Архитектура: 7 специализированных агентов

Система построена на базе **LangGraph StateGraph** — каждый агент является узлом графа, а оркестратор **Hermes** маршрутизирует задачи между ними. Все LLM-вызовы проходят через единый **LLMEngine** с поддержкой профилей (`dev_linux` / `mac_studio`).

| Агент | Роль | Модель (Mac Studio) | Ключевая специализация |
|-------|------|---------------------|----------------------|
| **Hermes** | Оркестратор / PM | Llama 3.3 70B 4-bit | Маршрутизация, вердикты, гибридная 3-стадийная модель |
| **ПТО** | Инженер ПТО | Gemma 4 31B 4-bit (VLM) | ВОР, чертежи, спецификации, Vision, OCR |
| **Юрист** | Юридическая экспертиза | Gemma 4 31B 4-bit (shared) | БЛС (58 ловушек), контракты, риски, претензии |
| **Сметчик** | Сметный расчет | Gemma 4 31B 4-bit (shared) | ФЕР/ТЕР, НМЦК, рентабельность |
| **Закупщик** | Тендеры и закупки | Gemma 4 31B 4-bit (shared) | Поиск лотов, НМЦК, стратегия торгов |
| **Логист** | Снабжение | Gemma 4 31B 4-bit (shared) | Поставщики, цены, КП, поставки |
| **Делопроизводитель** | Документооборот | Gemma 4 E4B 4-bit | Регистрация, классификация, ИД |

> Gemma 4 31B — общая модель в памяти для ПТО, Юриста, Сметчика, Закупщика и Логиста (Shared Memory, 128K контекст). Одна копия (~23 GB) обслуживает 5 агентов. Qwen3.5-27B удалена из модельного ряда v11.3.

## Два режима работы

### 1. Поиск подходящих лотов
Мониторинг тендерных площадок → оценка НМЦК → извлечение ВОР → поиск поставщиков → сметный расчет → юридическая экспертиза → вердикт о участии.

### 2. Сопровождение строительства
Вводный контроль → оперативный контроль объемов → финансовый контроль (КС-2/КС-3) → логистическое обеспечение → претензионная работа → завершение объекта.

## Технологический стек

| Слой | Технология | Назначение |
|------|-----------|------------|
| **Оркестрация** | LangGraph StateGraph | Управление потоками между агентами |
| **LLM Engine** | LLMEngine (MLX + Ollama) | Унифицированный интерфейс с профилями |
| **LLM-модели** | Llama 70B, Gemma 4 31B, Gemma 4 E4B | Специализированные агенты |
| **Vision/OCR** | Gemma 4 31B (MLX-VLM, нативный VLM) | Анализ чертежей, скан-копий, OCR |
| **Embeddings** | bge-m3 (1024 dim) через Ollama | Векторный поиск (RAG) |
| **БД основная** | PostgreSQL 16 + pgvector | Документы, ВОР, БЛС, прайс-листы |
| **БД графовая** | NetworkX (in-memory + файловая сериализация) | Связи документов и норм |
| **Кэш** | In-process (cachetools) | Кэш состояний и результатов (Redis удалён) |
| **Поиск** | Гибридный RAG (Vector + Graph + RRF) | Семантический + контекстуальный поиск |
| **Wiki** | Obsidian (Markdown) | Правила агентов, самообучение |
| **Документы** | python-docx, PyMuPDF | Генерация DOCX, парсинг PDF |

## Ключевые компоненты

### LLMEngine с профилями
Единая точка входа для всех LLM-операций. Переключение между Ollama (разработка на Linux) и MLX (продакшен на Mac Studio) через переменную окружения `ASD_PROFILE`:

```bash
export ASD_PROFILE=dev_linux   # Linux + Ollama
export ASD_PROFILE=mac_studio  # Mac Studio + MLX
```

### БЛС — Библиотека Ловушек Сторон
**58 структурированных ловушек** из судебной практики и Telegram-каналов в **10 категориях**. Хранится в PostgreSQL с векторными эмбеддингами (pgvector), что позволяет Юристу осуществлять семантический поиск аналогичных ситуаций через RAG с весовым ранжированием.

| Категория | Кол-во | Вес | Описание |
|-----------|--------|-----|----------|
| payment | 8 | 100 | Ловушки оплаты |
| penalty | 7 | 100 | Необоснованные штрафы и пени |
| acceptance | 7 | 100 | Ловушки приёмки работ |
| scope | 7 | 100 | Изменение объёмов работ |
| warranty | 6 | 90 | Гарантийные удержания и обязательства |
| subcontractor | 5 | 100 | Ограничения на субподряд |
| liability | 6 | 100 | Безлимитная ответственность |
| corporate_policy | 5 | 90 | Корпоративные политики заказчика |
| termination | 4 | 100 | Расторжение договора |
| insurance | 3 | 90 | Страхование |

Источники пополнения: seed YAML (`traps/default_traps.yaml`), Telegram-каналы (17 каналов в каталоге `config/telegram_channels.yaml`), LLM-экстракция из судебной практики.

### Hermes Router — гибридная 3-стадийная модель решений
1. **Weighted Scoring:** Агентные веса (Юрист 0.35, Сметчик 0.25, ПТО 0.20, Закупщик 0.12, Логист 0.08). Score ≥ 0.7 → GO, ≤ 0.3 → NO GO.
2. **LLM Reasoning:** Для серой зоны (0.3–0.7) — Llama 3.3 70B принимает решение с обоснованием.
3. **Veto Rules:** Автоматический NO GO при: legal=DANGEROUS, margin<10%, critical_traps≥3, НМЦК<70%.

### Map-Reduce для анализа контрактов
Длинные контракты (100+ страниц, свыше 128K токенов) разбиваются на чанки (6000 tokens, overlap 300), каждый анализируется Юристом отдельно (map), затем результаты агрегируются (reduce). Для большинства контрактов (до 300K символов / ~128K токенов) используется Quick Review без Map-Reduce — полный контракт помещается в контекст Gemma 4 31B.

### Юридическая экспертиза (v11.3)
- **LegalService** — сервис юридической экспертизы с Map-Reduce и Quick Review
- **LegalAnalysisResult** — структурированный результат: findings (category, severity, clause_ref, legal_basis, issue, recommendation), verdict (approved/rejected/approved_with_comments), contradictions, normative_refs
- **VerdictReport** — итоговый вердикт с взвешенными сигналами агентов
- **ProtocolPartyInfo** — полные реквизиты сторон (ИНН, ОГРН, адрес, р/с, банк, БИК, подписант)
- **generate_protocol_docx()** — генерация протокола разногласий в DOCX с реквизитами
- **ст. 445 ГК РФ** — учёт принудительного заключения договора

### Цикл рефлексии (самообучение)
После каждого завершенного пайплайна система анализирует аудит-логи и обновляет wiki-правила агентов. Система умнеет с каждым обработанным лотом.

## Быстрый старт

### 1. Инфраструктура
```bash
cd infrastructure
docker compose up -d    # PostgreSQL
```

### 2. Настройка окружения
```bash
cp .env.example .env   # Указать ASD_PROFILE, пути к БД
```

### 3. Запуск системы
```bash
source .venv/bin/activate
export PYTHONPATH=$PYTHONPATH:.
export ASD_PROFILE=dev_linux    # или mac_studio
python src/main.py
```
